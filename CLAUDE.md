# GitHub State Mirror

## Project Overview

A Go service that mirrors GitHub state into SQLite, providing a fast local API surface. This is a **cache** — the SQLite DB is disposable and gets nuked+recreated on schema version changes.

## Architecture

- `internal/actor/` — Context-based actor (per-credential) identity propagation (stdlib-only, safe to import from anywhere)
- `internal/freshness/` — Generic cache freshness framework (zero GitHub knowledge)
- `internal/database/` — SQLite schema + sqlc-generated queries (`dbgen/` is codegen, do not edit)
- `internal/ghdata/` — Domain store wrapping sqlc with transaction logic (`dashboard.go` = cross-actor cache stats)
- `internal/ghclient/` — GitHub REST + GraphQL API client (includes in-memory token→login cache and `AppAuthenticator` for GitHub App JWT / installation-token minting)
- `internal/auth/` — GitHub OAuth login + signed-cookie sessions for the dashboard (stdlib-only, no DB knowledge)
- `internal/sync/` — Bridge layer: concrete fetchers, periodic refresh, webhook dispatch
- `internal/webhook/` — Webhook HTTP handler + event parsing (dispatch is **synchronous**; the handler returns a per-delivery disposition in the response so GitHub's delivery record reflects whether the cache was updated)
- `internal/api/` — chi router, REST handlers, GraphQL cache assembler, the GitHub passthrough proxy (`proxy.go`), the in-memory request-disposition log (`requestlog.go`), and the web dashboard (`dashboard.go` + embedded `web/`)
- `internal/api/web/` — Dashboard front-end: `src/*.ts` (TypeScript source) compiled to `assets/*.js` (committed, generated, embedded)
- `cmd/server/` — Entry point

## Key Constraints

- **No migrations** — bump `SchemaVersion` in `internal/database/db.go` to nuke+recreate
- **sqlc codegen** — run `sqlc generate` after modifying `schema.sql` or `queries/*.sql`
- **Freshness/data separation** — `internal/freshness/` must never import GitHub-specific packages
- **Per-credential cache isolation** — all cached data is scoped by actor, where the actor is a SHA-256 **fingerprint of the caller's token** (`ghclient.Fingerprint`), set in `requireAuth` (`internal/api/router.go`). Any valid GitHub token (PAT or App user-to-server) is accepted; isolation is by token, not token type. This is the security boundary: a credential only ever reads what it fetched, so the service is safe for untrusting multi-tenant use. The `actor` column is part of every table's primary key. Data endpoints reject tokenless requests with 401. There is **no static service token**: the **GitHub App** (`ghclient.AppAuthenticator`) is the service's only credential, used for background work (periodic refreshes and the webhook on-demand pull below), never to serve requests; the periodic refresher mints a short-lived access token per installation and runs in its own stable `app-installation:<id>` partition (see `sync.AppSessions`). Webhooks invalidate/apply across all actors via `MarkStaleByKindKey` / `ActorsForRepo` — `ActorsForRepo` is `SELECT DISTINCT actor FROM repos`, so a delivery for a repo no partition has cached would otherwise be `skipped` ("no cached scope"). Instead the dispatcher **pulls on demand**: `actorsForRepo` (`internal/sync/webhook.go`), finding zero actors, mints an installation token from the delivery's `installation.id` and `EnsureFresh`es org-repos for the owner into the `app-installation:<id>` partition, then re-queries — so the first webhook for a repo bootstraps a scope itself (one fetch seeds every repo the installation can see). This is best-effort: no app configured, no `installation.id`, or a failed fetch falls through to `skipped` (never `error`, to avoid GitHub retry storms). Do NOT replace this with proactive enumeration/seeding — partitions populate on demand (lazy fetch + this pull); the periodic refresher only `RefreshAllOfKind`s what a partition already holds, as a backstop.
- **Webhook-fed cache (the whole point)** — the dispatcher (`internal/sync/webhook.go`) applies payloads **directly** to the cache so high-frequency events never trigger a GitHub re-fetch: `pull_request`/`pull_request_review` upsert the PR; `status`/`check_run`/`check_suite` aggregate per-check state in the `commit_checks` table and roll it up onto each PR's `last_commit_status` (and the repo's `default_branch_status` when the check ran on the default branch, matched by `head_branch`/`branches` vs the payload's `default_branch`); `push` updates `pushed_at`; `label` recolors/removes labels. Invalidation (`MarkStaleByKindKey`) is only a **fallback** for structural events (`repository`/`organization`/`membership`) or unparseable payloads. Do NOT regress this into invalidate-and-refetch — leveraging pushed webhook state to avoid upstream fetches is the entire purpose of the project. `UpsertPullRequest` COALESCEs `last_commit_status` so a PR webhook (which carries no CI state) doesn't clobber a status set by a check webhook. The handler dispatches **synchronously** (the cache writes are fast, idempotent upserts) and returns a `DispatchResult` whose disposition — `applied` (200) / `skipped`·`ignored`·`invalidated` (202) / `error` (500) — is the response GitHub records; every delivery is also written to the global `webhook_deliveries` log. Don't restore the old fire-and-forget 200: it made every delivery look successful regardless of whether data was preserved.
- **App-identity partition (opt-in, for trusted first-party app callers)** — a *request*-time caller may send a **GitHub App JWT** in the `X-Mirror-Identity` header. `requireAuth` verifies it via `ghclient.VerifyAppIdentity` (`GET /app` — GitHub only 200s if the RS256 signature checks out against the app's public key, so it's unforgeable; cached per-JWT) and partitions that caller as `app:<id>` instead of by token fingerprint. This is distinct from the background refresher's `app-installation:<id>` partitions above: that's the mirror signing in as *its own* app for periodic refreshes; this is an *external* app caller (e.g. pr-minder) tagging its data-API requests. It exists because such a caller's installation tokens rotate hourly — fingerprinting them would give a fresh empty bucket every hour; an app identity gives **one stable, webhook-fed bucket** for all the app's tokens. The `Authorization` token is still injected into the context for upstream fetches/passthrough, so per-repo authorization is unchanged. Security caveat: within an app's bucket, any of that app's tokens can read what the bucket cached (the reader is a single trusted codebase), so this is appropriate **only** for a first-party app, never as a way to relax the default fingerprint isolation. No identity header → unchanged fingerprint behavior.
- **Identical-or-passthrough (the cache contract)** — a route is served from cache ONLY if its response is **byte-for-byte identical to GitHub's**; otherwise it must passthrough. The cache is not a transformative middleman. The mirror used to cache `/user`, `/user/orgs`, `/compare`, and `/pulls/{n}/files` with *trimmed* shapes — those routes were removed (`rest.go`/`router.go`), so the **only cached data route is `POST /graphql`** (the org-repos query, assembled in `graphql.go` to match exactly GitHub's selection set, locked by an identity test). Everything else is transparently reverse-proxied to GitHub **uncached** via `newGitHubProxy` (`internal/api/proxy.go`), wired as chi's `NotFound`/`MethodNotAllowed` (so the `Use` middleware — CORS, recoverer — still wraps it); the GraphQL handler forwards any non-org query the same way. The proxy forwards the caller's `Authorization` verbatim (never the server token), rejects tokenless requests with 401, targets `gh.BaseURL()` (tests hit the fake upstream), and `ModifyResponse` strips GitHub's duplicate `Access-Control-Allow-*` while keeping `Access-Control-Expose-Headers`. Do NOT make this path touch the freshness store. A cached endpoint may be added back only with a GitHub-faithful shape **and** an identity test (`identity_shape_test.go` / passthrough tests in `proxy_test.go`); otherwise leave it passthrough.
- **OAuth token-exchange relay** — `POST /login/oauth/access_token` (`internal/api/oauth.go`) relays a GitHub OAuth code-for-token exchange to `github.com` (NOT `api.github.com`) and returns it with the mirror's CORS headers, so a purely client-side app can complete login (GitHub's token endpoint sends no CORS headers). It is registered outside `requireAuth` (the OAuth `client_secret` in the body is the credential — no bearer token) and forwards only the body + content-negotiation headers to a fixed URL (`githubOAuthTokenURL`, a var so tests can point it at a fake). It does NOT copy GitHub's `Access-Control-*` (corsMiddleware is the single CORS authority, so no duplicate ACAO). This is why repo-nightmare's `TOKEN_PROXY` points here instead of a standalone CORS proxy.
- **Dashboard = separate authz model** — the web dashboard (`internal/api/dashboard.go`, served at `/`) authenticates a human via **GitHub OAuth** and authorizes by **login** (session cookie), which is deliberately distinct from the data API's bearer-token + fingerprint model. For a normal user it never serves one credential's cached rows to another — it only reports per-scope **counts + freshness metadata**. A user sees the scopes their own tokens populated; logins in `ADMIN_LOGINS` (default `PazerOP`) see all scopes. The `actor_identities` table maps fingerprint→login for this grouping only; it does NOT relax data isolation (data tables stay keyed by the opaque fingerprint — do NOT switch the data partition to a username/hash, or a narrow token would read what a broad token cached). Identity rows are written (debounced) in `requireAuth`. Admins additionally get two global activity logs: a **webhook delivery log** (`GET /api/webhooks`, the `webhook_deliveries` table, "Webhooks" tab) showing each delivery's disposition + detail, and a **request-activity log** (`GET /api/requests`, the in-memory `requestLog` in `requestlog.go`, "Requests" tab) showing each data-API request's cache disposition — `hit` (served fresh) / `miss` (fetched then cached) / `passthrough` (forwarded uncached) — plus per-disposition totals, so you can see how much the cache is actually used vs. proxied through. Disposition comes from `freshness.Manager.EnsureFreshOutcome` for the cached `/graphql` route and from the `recordPassthrough` wrapper around `newGitHubProxy` for everything forwarded; the actor label is best-effort (the resolved actor, else the `X-Mirror-Identity` app id, else a token fingerprint). Both logs span every repo and are **admin-only**; the request log is **in-memory** (request traffic is high-volume and this is a live view, not an audit log) so it resets on restart.
- **Admin cache browse + consistency check (admin-only operator tooling)** — two admin-gated endpoints (`requireAdmin` in `internal/api/browse.go`, surfaced as per-scope **Browse** / **Check** buttons in the dashboard) let the operator inspect a scope beyond counts. `GET /api/cache/data?actor=<fp>` dumps the actual cached rows for one scope as flattened JSON (`collectBrowse` + the `*ByActor` reads in `internal/ghdata/browse.go` / new `dashboard.sql` queries). `GET /api/cache/check?actor=<fp>[&org=]` runs the **consistency check** (`internal/sync/consistency.go`): it re-fetches the source of truth from GitHub via the mirror's **GitHub App** (`AppAuthenticator`, shared with background refresh — `buildAppAuthenticator` in `main.go` builds it once and feeds both the periodic `SessionFunc` and the `ConsistencyChecker`) and emits a JSON diff (repos/PRs only-in-cache / only-on-github, plus field mismatches like `last_commit_status`, `default_branch_status`, labels, draft). The check is **read-only** (never writes the cache), only compares **OPEN** PRs (the cache only retains open PRs), and only checks owners with an Organization-type App installation (others go under `orgs_skipped`, not reported as missing). Returns **503** when no App is configured. This does NOT relax fingerprint isolation — it just lets the operator read what is already cached and diff it; it is gated to `ADMIN_LOGINS`. `scopeStats.actor_id` carries the full fingerprint so the admin UI can target a scope (display still uses the short form).
- **TypeScript front-end** — the dashboard JS is authored in `internal/api/web/src/*.ts` and compiled by `npm run build` (tsc) to `internal/api/web/assets/*.js`, which is **committed as a generated artifact** and embedded via `//go:embed`. Edit the `.ts`, never the `.js`. CI's `web-check` job fails if the committed JS is stale (run `npm run build` and commit). `assets/demo-data.js` is preview-only (NOT embedded); the CI `preview` job injects it to deploy a backend-free styling preview to buildhost per branch. Each `src/*.ts` emits its own standalone ES module loaded by its own `<script type="module">` tag in `index.html` — `rate-meter.ts` self-registers the `<rate-meter>` web component (shadow DOM owns the tile's markup/styles; themed via the inherited `:root` custom properties) that the rate-limit tab's tiles are built from. Adding an asset file means wiring it in **three** places: the `//go:embed` + hashed-URL machinery in `internal/api/dashboard.go`, the `index.html` script tag, and the `preview` job's copied-assets list in `ci.yml`.
- **Build** — use `go-toolchain` (not `go build`/`go test` directly); run `npm run build` after editing `web/src/*.ts`

## Commands

- `go-toolchain` — runs mod tidy, vet, test, build (all-in-one)
- `sqlc generate` — regenerates `internal/database/dbgen/*.gen.go` (after editing `schema.sql` or `queries/*.sql`)
- `npm run build` — compiles `internal/api/web/src/*.ts` → `internal/api/web/assets/*.js` (after editing the dashboard front-end)

## Environment Variables

- `GITHUB_APP_ID` (optional) — GitHub App ID. When set with a private key, the service signs in as this app for background work — periodic refreshes, the webhook dispatcher's on-demand repo pulls, and the source-of-truth fetch for the admin consistency check (`/api/cache/check`); it is never used to serve normal data-API requests, which require the caller's own `Authorization` header (401 otherwise). Unset = periodic refreshes and on-demand pulls disabled (webhooks for uncached repos skip) and the consistency check reports "unavailable" (503).
- `GITHUB_APP_PRIVATE_KEY` / `GITHUB_APP_PRIVATE_KEY_PATH` — the App's PEM private key (PKCS#1 or PKCS#8), inline (optionally `\n`-escaped) or as a file path (path wins). Required when `GITHUB_APP_ID` is set.
- `WEBHOOK_SECRET` — GitHub webhook HMAC secret
- `LISTEN_ADDR` — HTTP listen address (default `:8080`)
- `DB_PATH` — SQLite database file path (default `github-mirror.db`)
- `ALLOWED_ORIGINS` (optional) — comma-separated CORS allow-list for browser clients (e.g. the repo-nightmare PR viewer). Defaults to `*` (any origin), which is safe because data is isolated by token fingerprint, not origin. Preflight `OPTIONS` is answered without auth; see `internal/api/cors.go`.
- `GITHUB_OAUTH_CLIENT_ID` / `GITHUB_OAUTH_CLIENT_SECRET` (optional) — GitHub OAuth App credentials for dashboard login. If unset, the dashboard still renders but sign-in is disabled. Register the OAuth App's callback as `<BASE_URL>/auth/callback`.
- `SESSION_SECRET` (optional) — HMAC key for session cookies. If unset, a random per-process key is used (sessions reset on restart); set it in production.
- `ADMIN_LOGINS` (optional) — comma-separated logins that may view **all** cache scopes (default `PazerOP`); case-insensitive.
- `BASE_URL` (optional) — public base URL (e.g. `https://github-state-mirror.pazer.io`) used to build the OAuth `redirect_uri`; derived from the request (honoring `X-Forwarded-Proto`) when unset.
