# GitHub State Mirror

## Project Overview

A Go service that mirrors GitHub state into SQLite, providing a fast local API surface. This is a **cache** — the SQLite DB is disposable and gets nuked+recreated on schema version changes.

## Architecture

- `internal/actor/` — Context-based principal-key propagation + display shortening + verified display-name propagation (`WithName`/`NameFromContext`, set only from GitHub-verified answers: a user's login, an app's slug, an installation's account login) (stdlib-only, safe to import from anywhere). A "principal" identifies WHO is asking (`user:<id>`, `app:<id>`, `app-installation:<id>`, or a token fingerprint) — it gates what the cache REVEALS, never what it STORES; the name is display-only and never a key.
- `internal/freshness/` — Generic cache freshness framework (zero GitHub knowledge); `Manager.Drain` waits out in-flight detached fetches at shutdown
- `internal/database/` — SQLite schema + sqlc-generated queries (`dbgen/` is codegen, do not edit)
- `internal/ghdata/` — Domain store wrapping sqlc with transaction logic: global truth writes (`SyncOrgTruth`, `UpsertPRWithChecks`, ...), the reveal layer's `access_grants`/`deny_cache`, and the dashboard reads. The grant-writing paths (`SyncOrgTruth`, `RecordGrant`) opportunistically sweep expired grants/denials after committing (`maybePruneAccessControl`: best-effort, log-on-error, throttled to once per ~10 min via an atomic last-run stamp); reads (`HasGrant`, `GrantsByPrincipal`, `CountLiveGrants`, `ListGrantPrincipals`) all filter on expiry regardless
- `internal/ghclient/` — GitHub REST + GraphQL API client (includes the in-memory token→identity cache behind `ResolveTokenIdentity` and `AppAuthenticator` for GitHub App JWT / installation-token minting; `Installations` paginates past 100). Two GraphQL fetch families: the identity-locked `GetOrgData` (`organization(login:)`, the lazy /graphql route's byte-frozen contract) and the sync/checker-private `GetOwnerData` + `OwnerRepoVisibilities` (`repositoryOwner(login:)` — resolves Users AND Organizations; selects extras the locked query must never grow: `isArchived`, `labels(first: 100)`, `autoMergeRequest`, the `visibility` enum). Both owner queries pin **`ownerAffiliations: OWNER`** — the connection's default `[OWNER, COLLABORATOR]` made a User's listing include repos the login merely *collaborates* on, under their real owners, which `convertRepo`/`convertPR` then keyed by the QUERY login (the collaborator-repo bleed: junk `<user>/<name>` truth rows with foreign `name_with_owner`/`url`, double-counted PRs). Belt-and-braces, every repo-conversion loop (org and owner alike, plus the visibility loop) also **DROPS any node whose real owner** (`nameWithOwner` prefix, else `owner.login`) **differs case-insensitively from the queried login**, logging each drop (`dropForeignRepoNode` in `queries.go`) — drop, never re-key, so one owner's sync can never absorb truth rows or mint grants across owners; this also covers the identity-locked org query with zero query-text change (`TestOrgQueryUntouched` still passes) and kills the visibility map's bare-name collision hazard (a same-named foreign node clobbering an owned repo's visibility entry). SchemaVersion 14's deploy-time nuke cleaned up the junk rows a corrected re-run could never delete (sync never deletes repo rows; the PR reconcile only visits fetched repos). `SetRateObserver` installs the passive rate-limit hook: every response the client sees is reported to `internal/ratemeter`, labeled by the ctx principal when set, else by the credential's shape (`app-jwt` for a JWT, a short `token:<fp12>` fingerprint otherwise — never the raw token)
- `internal/ratemeter/` — Passive rate-limit observation: a thread-safe, IN-MEMORY store of the latest `X-RateLimit-*` reading per (identity, resource), fed by every upstream path (passthrough proxy, cached-route miss fetches, reveal probes, ghclient calls) and read by the dashboard's "Rate limit" tab. Each observation also carries the identity's verified display `Name` when the caller supplied one (display-only, never part of the key; a nameless re-observation keeps the last known name). Deliberately in-memory (the `requestLog` stance: live view, not an audit log — and a DB table would force a cache-nuking SchemaVersion bump for sub-hour-ephemeral data); bounded at 512 entries (oldest-ObservedAt evicted); methods are nil-receiver-safe; resets on restart
- `internal/reqtimeline/` — In-memory ring of TIMED traffic events behind the dashboard's "Timeline" chart: incoming webhook deliveries (receipt→dispatch-complete) and outgoing proxied GitHub requests (upstream round-trip), every event carrying its REAL measured duration (never faked to an instant, never inflated). Fed by exactly three seams — the webhook handler's `DeliveryRecorder`, `recordPassthrough`, and respcache's `fetchUpstream` (ghclient's own background traffic is deliberately out of scope: not "through the proxy") — and read by the admin-only `GET /api/timeline?since=<id>` cursor endpoint. Bounded by 24h retention (lazy eviction on write AND read; no background timers) plus a 100k count cap; nil-receiver-safe; in-memory on the requestLog/ratemeter stance (live view, not an audit log; resets on restart). Lanes are bounded by construction: "⇐ <event type>" for webhooks, `normalizeRoute`'s route shapes for requests — never per-URL.
- `internal/auth/` — GitHub OAuth login + signed-cookie sessions for the dashboard (stdlib-only, no DB knowledge)
- `internal/sync/` — Bridge layer: the org list-sync fetcher, periodic refresh, webhook dispatch (always-apply), the global consistency checker
- `internal/webhook/` — Webhook HTTP handler + event parsing (dispatch is **synchronous**; the handler returns a per-delivery disposition in the response so GitHub's delivery record reflects whether the cache was updated). After each dispatch the handler hands the outcome to any `IngestNotifier` (variadic, optional, nil-safe — `internal/notify` implements it); the notifier call is non-blocking by contract, so the response to GitHub never waits on subscribers. An optional `DeliveryRecorder` (nil keeps it inert; `internal/api` adapts it onto `reqtimeline`) additionally observes every VERIFIED delivery with its real measured handling duration for the dashboard's Timeline chart — unverified deliveries never leave a trace.
- `internal/notify/` — Subscriber notifications ("mirror-ingest webhooks"): authenticated API clients register their own webhook endpoints (`/_mirror/subscriptions` CRUD, principal-scoped via requireAuth), and AFTER the dispatcher applies a delivery to global truth the mirror POSTs a compact, HMAC-signed JSON notification (`X-Hub-Signature-256`, GitHub's exact scheme) to every matching, authorized subscription — so consumers stop racing the mirror's ingestion with their own GitHub webhooks. Subscriptions persist in a SEPARATE SQLite config DB (`SUBSCRIPTIONS_DB_PATH`); delivery is detached, bounded, retried, auto-disabling, and reveal-gated fail-closed (see the doctrine bullet under Key Constraints). In-memory activity ring + counters back the admin-only `GET /api/notifications` JSON view (no dashboard tab).
- `internal/api/` — chi router, REST handlers, GraphQL cache assembler, the **reveal layer** (`reveal.go`), the GitHub passthrough proxy (`proxy.go`), the in-memory request-disposition log (`requestlog.go`, with per-route-shape grouping in `requestgroups.go`), and the web dashboard (`dashboard.go` + embedded `web/`)
- `internal/api/web/` — Dashboard front-end: `src/*.ts` (TypeScript source) compiled to `assets/*.js` (committed, generated, embedded)
- `cmd/server/` — Entry point (drains in-flight fetches before closing the DB on shutdown)

## The global-truth model (operator directive, 2026-07-03)

> "just because nobody has fetched something doesn't mean we get to ignore updates from webhooks for it. there's only 1 cache because there's only 1 true state, the appropriate parts of the cache are revealed to users/apps based on their permission model (i.e. were they able to access this resource directly through github -> yes -> ok we can serve the cache for this resource to them)"

Storage and authorization are **separate axes**:

1. **ONE global truth store.** `repos`, `pull_requests`, `pr_labels`, `commit_checks`, `contents_cache`, `git_commits_cache`, `git_commit_miss_cache`, `pulls_list_cache`, `commits_list_cache`, `compare_cache`, `commit_ci_cache`, `workflow_runs_cache`, `pull_files_cache`, `closed_pull_cache`, `pull_diff406_cache`, `branches_list_cache` carry NO actor column — there is one row per fact because there is one true state. Webhooks and every principal's fetches all maintain the same rows. (`install_token_cache` and `repo_installation_cache` stay keyed by the app principal: a minted token IS per-credential, not shared truth.)
2. **Webhooks ALWAYS apply.** The dispatcher never asks "who has this cached?" — a delivery for a repo nobody ever fetched upserts the `repos` row straight from the payload's own `repository` object (`absorbRepoFromPayload`) and applies. There is no `skipped` disposition, no on-demand pull, and no `ActorsForRepo` gate; those were deleted.
3. **Reads reveal by permission** (`internal/api/reveal.go`). Serving a cached resource requires proof the caller could read it from GitHub directly: the repo is **public** in truth (fast path, no GitHub call), the principal holds a live **access grant**, or a live **probe** (`GET /repos/{owner}/{repo}` with the caller's own token) answers 2xx. Grants are earned only from GitHub's own answers — an org list-sync **replace-syncs** the principal's grants to exactly the repos GitHub returned them (`source=list_sync`), and a successful probe records one (`source=probe`); TTL ~24h (`ghdata.GrantTTL`). Authoritative denials (404, non-rate-limit 403) are cached in `deny_cache` for ~5m (`ghdata.DenyTTL`) and replayed without touching GitHub; transient failures (5xx, 429, rate-limited 403, network) are NEVER cached and fail only that one request. Unknown visibility (`repos.visibility` = `''`) is treated as private — fail closed.

## Key Constraints

- **No migrations** — bump `SchemaVersion` in `internal/database/db.go` to nuke+recreate
- **sqlc codegen** — run `sqlc generate` after modifying `schema.sql` or `queries/*.sql`
- **Freshness/data separation** — `internal/freshness/` must never import GitHub-specific packages
- **Lazy fetches are detached, backoff-gated, and drained** — `freshness.Manager.doFetch` runs the fetch and its metadata writes on a `context.WithoutCancel`-derived context bounded only by `fetchSafetyTimeout` (5m): a client aborting its request cannot kill the shared multi-page org-repos fetch mid-flight or prevent the result from being stored, and a short caller deadline is deliberately NOT re-applied (the old webhook-deadline re-application killed fetches half-done). Every detached fetch is tracked in an inflight WaitGroup; `Manager.Drain(timeout)` waits for them and `cmd/server` calls it after `ListenAndServe` returns, BEFORE `db.Close()` — so a fetch can never write to a closed DB at shutdown. An error-state row inside its `retry_after` window is NOT re-fetched by lazy callers — `EnsureFreshOutcome` reports the stored error instead, so a failing upstream isn't hammered on every request; deliberate refreshes (periodic triggers) bypass the backoff. When `/graphql` serves the stale cache after a refresh failure it sets `X-GSM-Stale: true` + `X-GSM-Last-Fetched` response headers and WARN-logs with the principal — headers only, the body stays identity-test-identical.
- **One global truth, revealed by permission (replaces per-actor isolation)** — see "The global-truth model" above; this is the load-bearing security section. The principal is resolved in `requireAuth` (`internal/api/router.go`): a user token resolves via `ghclient.ResolveTokenIdentity` (`GET /user`, cached per token) to `user:<numeric id>` (stable across renames, shared by all of that user's tokens); a token that is definitively NOT a user (non-rate-limit 403 / 404 on `/user`) falls back to the SHA-256 token fingerprint; a transient `/user` failure with no cached verdict fails 503 (never a silent guess). The security boundary is **the reveal layer, not storage**: one user can never read another's private-repo data because revealing `owner/repo` to a principal requires the repo to be public, a live grant (earned from GitHub answering THAT principal), or a live probe with THAT principal's own token. Public repos are served to any authenticated caller. Fail-closed rules that must not regress: unknown visibility = private; deny verdicts cache only authoritative answers (a rate-limited 403 must never pin a legitimate caller out for the TTL); grants expire (~24h) so revoked GitHub access converges; `SyncOrgTruth` REPLACE-syncs list_sync grants (a repo GitHub stopped returning loses its grant). Data endpoints reject tokenless requests with 401. There is **no static service token**: the **GitHub App** (`ghclient.AppAuthenticator`) is the service's only credential, used for background work (periodic refreshes and the consistency check), never to serve requests; the periodic refresher mints a short-lived access token per installation and runs as the stable `app-installation:<id>` principal (see `sync.AppSessions`), so its syncs both refresh global truth and hold that principal's grants. The refresher syncs each installation's ACCOUNT by name (`Session.Owner` → `InvalidateAndRefresh`, which seeds the freshness row itself — a fresh installation is synced on the first cycle) via the owner-agnostic `GetOwnerData`; the fetcher picks the query by principal, so lazy callers keep the identity-locked org query. (The old `RefreshAllOfKind` shape only re-fetched rows that already existed under the actor — and nothing ever created one, so fleet sync silently never ran.)
- **Webhooks always apply (the whole point)** — the dispatcher (`internal/sync/webhook.go`) applies payloads **directly** to global truth so high-frequency events never trigger a GitHub re-fetch: `pull_request`/`pull_request_review` upsert the PR (and upsert the repos row from the payload first, so a never-fetched repo is absorbed on its first delivery); `status`/`check_run`/`check_suite` aggregate per-check state in the `commit_checks` table and roll it up onto each PR's `last_commit_status` (and the repo's `default_branch_status` when the check ran on the default branch); `push` updates `pushed_at`, absorbs the pushed commits into `git_commits_cache`, **un-resolves `mergeable`** on every open PR whose base or head is the pushed branch (`NullPRMergeableByBranch` — GitHub recomputes mergeability after either side moves and never webhooks the result), and — when the pushed ref IS the default branch — **un-resolves `default_branch_status`** the same way (the stored rollup describes the previous tip; a tip with no CI would otherwise keep it forever since the COALESCE upsert can never clear it; the next default-branch check event repopulates); `label` recolors/removes labels; `repository` handles lifecycle (deleted → `DeleteRepoCascade`, renamed → delete the old name + absorb the new, privatized/publicized → `SetRepoVisibility`, other actions → absorb or invalidate). Invalidation (`MarkStaleByKindKey`) is only a **fallback** for structural events with unparseable payloads. Do NOT regress this into invalidate-and-refetch, and do NOT add a "who has this cached?" gate — truth is maintained unconditionally. `UpsertPullRequest` COALESCEs `last_commit_status` AND `mergeable` against the existing row, so a PR webhook — which carries no CI state, and carries `mergeable: null` while GitHub is still computing it — doesn't clobber known values (the un-resolve signal is the PUSH event, not the PR event's transient null); a genuinely resolved `mergeable` still overwrites. GraphQL-sourced REST-only columns (`node_id`, `body`, `auto_merge_method`, `merge_commit_sha`, ...) are guarded the same way so a GraphQL sync can't blank what a webhook/REST absorb recorded. `UpsertPRWithChecks` additionally derives `last_commit_status` from the commit checks already recorded for the PR's head sha, so a PR opened *after* its head commit's CI finished (pr-minder auto-opened PRs) doesn't sit at NULL forever. The handler dispatches **synchronously** and returns a `DispatchResult` whose disposition — `applied` (200) / `ignored`·`invalidated` (202) / `error` (500) — is the response GitHub records (there is NO `skipped`); every delivery is also written to the global `webhook_deliveries` log. Don't restore the old fire-and-forget 200. The dispatcher also maintains the response caches (disposition-neutral, best-effort; round 2 made the grain **per-ref** wherever the payload names the moved refs — `invalidateResponseCaches`/`invalidateForPush`): a `push` flushes the PUSHED REF's rows in the ref-keyed tables — `contents_cache` and `commits_list_cache` per ref, PLUS their empty-ref spelling when the pushed ref IS the payload's `default_branch` (an empty ref key means "the default branch", so a default-branch push moves both spellings; a payload that doesn't say which branch is the default falls back repo-wide for exactly these two tables), `compare_cache` rows naming the ref on EITHER side (`base_ref`/`head_ref` — a 404 unknown-ref verdict clears the moment a push creates the ref), and `commit_ci_cache` rows keyed by the ref's verbatim spelling — while `pull_files_cache`, `branches_list_cache`, and `pull_diff406_cache` stay REPO-WIDE on every push (no per-ref signal: a base push moves merge-base-relative file lists and can move any PR's three-dot diff across the 406 boundary, and any push edits the branch listing); a push payload with no usable ref keeps the old conservative whole-repo flush for all the ref-keyed tables. Every per-ref branch/tag flush covers the ref in EVERY spelling GitHub accepts (`refSpellings`: the bare name plus `heads/<name>`/`refs/heads/<name>`, or the tags forms) — rows key the verbatim requested spelling, so a bare-name-only flush would leave the qualified spellings serving stale; sha flushes stay single-spelling (an abbreviated-sha-keyed row is bounded by the 24h TTL, and surveyed consumers send full hex). `status`/`check_run`/`check_suite` flush per the refs the payload names — the head branch(es) + the sha itself for `commit_ci_cache` (rows key the verbatim requested ref, so branch-form and sha-form rows both go) and the sha's `workflow_runs_cache` pages — falling back repo-wide for both when the payload names none. `workflow_job` deliveries flush the job's `head_sha`'s `workflow_runs_cache` pages (repo-wide when absent) — and this runs for EVERY delivery, BEFORE the disposition logic, so the `queued`/`waiting` jobs the apply side drops as `ignored` still flush (a queued job is a run the cached listing may not have shown yet). `workflow_run` deliveries flush the run's `head_sha`'s `workflow_runs_cache` pages the same way (repo-wide when absent) — the ONLY signal for a `startup_failure` run, which creates no jobs, check runs, or statuses; the truth side has no `workflow_run` handler, so the delivery still records as `ignored` (invalidation precedes disposition, the queued-workflow_job precedent). `pull_request`/`pull_request_review` events flush that ONE PR's `pull_files_cache` pages, its `closed_pull_cache` doc, and its `pull_diff406_cache` verdict (head pushes — including fork heads whose push events never arrive — base retargets, reopens; PRNumber-guarded, any action). `repository` flushes EVERYTHING repo-wide: all of the above plus the `pulls_list_cache` marker, the `closed_pull_cache` docs, and the `git_commit_miss_cache` markers (`pull_request` events deliberately do NOT touch the list marker — they maintain the PR rows the marker vouches for; and a `push` deliberately never touches closed docs — it cannot mutate a closed PR). Positive `git_commits_cache` rows are immutable and are NEVER invalidated; the git-commit 404 MISS markers are instead cleared by the absorb path itself (every real commit upsert un-misses its sha — `ghdata.upsertGitCommit`, the single funnel all absorb paths share). Unparseable payloads always fall back to the repo-wide flush. `installation`/`installation_repositories` flush the installation's `install_token_cache` AND `repo_installation_cache` rows. `workflow_job` deliveries (`in_progress`/`completed` only; `queued`/`waiting` dropped as `ignored` — after the cache flush above) record job state in the global `workflow_jobs` table (admin-only `GET /api/jobs`; out-of-order tolerant upsert; completed rows pruned after 14 days). **Operator action required**: the GitHub App must be subscribed to the `workflow_job` event for that table to fill — and to the `workflow_run` event for the startup-failure `workflow_runs_cache` flush above (until it is, that staleness window is bounded by the cache's 24h TTL).
- **App-identity principal (opt-in, for trusted first-party app callers)** — a *request*-time caller may send a **GitHub App JWT** in the `X-Mirror-Identity` header. `requireAuth` verifies it via `ghclient.VerifyAppIdentity` (`GET /app` — GitHub only 200s if the RS256 signature checks out against the app's public key, so it's unforgeable; cached per-JWT) and resolves that caller to the principal `app:<id>`, skipping the per-user/fingerprint resolution entirely (installation tokens cannot call `/user`). This exists because such a caller's installation tokens rotate hourly — fingerprint principals would lose their earned grants every hour; the app principal keeps **one stable grant set** across all the app's tokens. The `Authorization` token is still injected into the context for upstream fetches/probes/passthrough, so per-repo authorization against GitHub is unchanged. No identity header → the default per-user/fingerprint resolution above.
- **The cache contract (three tiers)** — every data route falls into exactly one of these; know which tier a route is in before touching it:
  1. **GraphQL org-repos — byte-identical, identity-test-locked (unchanged).** `POST /graphql`'s org-repos query is assembled in `graphql.go` to match exactly GitHub's selection set, locked by an identity test (`identity_shape_test.go`). This is the ONLY route where byte-identity to GitHub applies. The assembly reads global truth **filtered by the reveal layer** (`ListVisibleReposByOwner`: public repos + the principal's granted repos), so the served view tracks what GitHub itself answers this principal — never the whole truth store. The backing fetch (`OrgReposFetcher`) runs with the PRINCIPAL'S OWN token, merges the snapshot into global truth (`SyncOrgTruth`: upsert + reconcile, never delete-then-insert), and replace-syncs the principal's grants. Known bounded caveat: the locked selection pages labels at `first: 10`, so a lazy sync stores a 10-label snapshot per PR (a >10-label PR is truncated on this path — the contract forbids extending the query; the App-driven owner query reads 100 and the apply mode corrects the stored set). Any other GraphQL query forwards to the passthrough (a `mutation` is recorded as a `write`).
  2. **Cached REST routes — global state absorbed, rebuilt TRIMMED, revealed by permission.** `internal/api/respcache*.go` (one file per route family: contents in `respcache.go`, then `_gitcommits`, `_commits`, `_commitci`, `_compare`, `_pulls`, `_pulldiff`, `_pullfiles`, `_branches`, `_repo`, `_installation`, `_actionsruns`) + `internal/ghdata/respcache*.go`. Every read passes the reveal gate first (`reveal(r, owner, repo, kind, key)`): public repo / live grant / cached deny verdict / live probe — a denied caller gets GitHub's own relayed answer, an unrevealable transient error 502s that one request, and a successful probe absorbs the probed repo object into truth and records a grant. The route ABSORBS the state a GitHub response contains into structured columns and REBUILDS the response from that state, **dropping every URL field**: `url`, anything matching `*_url`, and `_links` (locked by `assertNoURLKeys` in `respcache_test.go`) — EXCEPT a per-route **pinned allowlist** of consumer-read link fields (the helper takes exact key names; today: the statuses `target_url`, the check-run `details_url`/`html_url`, and the workflow-run `html_url`, all rendered by the required-builds hook — consumer survey 2026-07-11). Adding an allowlist entry requires a fresh consumer survey; every other URL key stays banned. Hits AND misses both serve the rebuilt shape, marked `X-GSM-Cache: hit|miss`; rebuilt responses carry only `Content-Type: application/json; charset=utf-8`. Anything a route cannot model passes through verbatim, unstored, recorded as `passthrough`: non-default `Accept` media types, unknown query params, symlink/submodule objects, >1 MiB `"encoding":"none"` files, oversized bodies, unexpected statuses. The v1 routes:
     - `GET /repos/{owner}/{repo}/contents/{path...}` (incl. `?ref=`, part of the key) — absorbs 200 file (base64), 200 dir (trimmed entries), **and 404** (the "config file absent" answer is half the win). One global row per (owner, repo, path, ref). Invalidated per PUSHED REF (+ the empty-ref default-branch spelling) by `push`, repo-wide by `repository` — see the dispatcher bullet's per-ref matrix — plus a **24h TTL backstop**.
     - `GET /repos/{owner}/{repo}/git/commits/{sha}` (`respcache_gitcommits.go`) — positive rows are immutable: no invalidation, no TTL, LRU-bounded only. Rebuild: `{sha, author{...}, committer{...}, message, tree:{sha}, parents:[{sha},...]}`. **Push payloads feed this cache too** (`absorbPushCommits`): each pushed commit is upserted once, globally; parents come from the payload's linear chain (`webhook.PushPayload.ChainedCommits`), which declines forced pushes, new refs, possibly-truncated arrays (≥20), and non-linear orderings rather than derive wrong parents. A **404 is cached as an expiring MISS MARKER** (`git_commit_miss_cache`, rebuilt in the contents route's `{"message":...,"status":"404"}` shape, 24h TTL) — a round-2 stance change: round 1 deliberately didn't cache these ("a missing sha can be pushed later"), but the dominant traffic is pr-minder's `mergeWouldBeEmpty` re-reading GC'd test-merge shas forever, each read a fresh upstream 404. Two bounds make it safe: the expiry, and the **clear-on-upsert invariant** — every real commit absorb (fetch, push payload, commits-list, compare) funnels through `ghdata.upsertGitCommit`, which clears the sha's marker, so a materialized sha stops answering 404 immediately; consumers fail open on 404 regardless.
     - `GET /repos/{owner}/{repo}/commits` — the commits LIST (behind reveal; pr-minder's fork-point base detection pages this per branch, dozens per sweep — the passthrough flood that motivated the route). Absorb-don't-byte-cache: each listed commit is upserted into the SAME global `git_commits_cache` rows above; a `commits_list_cache` snapshot per exact modeled query shape — key `(owner, repo, ref_param, per_page, page)`, where `ref_param` is the raw `sha` query value (`''` = default branch) — stores only the response's ordered shas (the rows+marker split the pulls list uses). A hit needs an unexpired snapshot AND every listed sha still resolving in `git_commits_cache` (an LRU-pruned commit degrades to a miss, self-healing; the re-absorb restores the pair). Hit and miss rebuild the identical trimmed item `{sha, commit:{author{name,email,date}, committer{...}, message, tree:{sha}}, parents:[{sha},...]}` — a superset of the one field the consumers (pr-minder `listCommitShas` + the reconcile hook's port) actually read (the top-level `sha`; they paginate by short-page detection, no Link header); the top-level author/committer USER objects, `node_id`, `verification`, `comment_count`, and every URL field stay dropped. Query-shape guard: only `sha` (non-empty), `per_page` (1..100, default 30), `page` (1..10, default 1) are modeled — anything else (`path`, `since`, `until`, `author`, ...) or a non-default `Accept` passes through. Unlike the pulls list there is NO full-page truncation guard: the snapshot IS that exact page's answer, and pagination continues under the next page's own key. Only a 200 absorbs (an empty array — a page past the end — is a valid cacheable answer); 404 (unknown ref), 409 (empty repo), 5xx relay unstored. Snapshots are flushed per PUSHED REF (+ the empty-ref default-branch spelling) by `push`, repo-wide by `repository`, + a 24h TTL backstop. The single-commit sub-path `/repos/{owner}/{repo}/commits/{sha}` (a different shape; pr-minder's `commitAgeSeconds`) deliberately stays passthrough — since the commit-CI routes below registered the `/commits/*` subtree, that forward now happens in the subtree dispatcher rather than the router's NotFound.
     - `GET /repos/{owner}/{repo}/compare/{basehead}` — the three-dot `base...head` comparison (behind reveal; pr-minder's `compareCommits` — the `auto_open_pr` open gate + `closeEmptyAutoPrs`, repeated fleet-wide every 5 minutes by the pr-minder-reconcile hook: the ~90%-of-request-log passthrough flood that motivated the route). HISTORY: an earlier /compare cache was REMOVED because its trim dropped `files`, breaking pr-minder's empty-PR gate — this rebuild keeps every consumer-read field: `{status, ahead_by, behind_by, total_commits, merge_base_commit:{sha}, commits:[<the commits-list trimmed item shape>], files:[{filename, status, additions, deletions, changes, previous_filename?}]}` with the **files array's presence/absence preserved exactly** (consumers read `changed_files = files.length`; ABSENT = unknown → fail open, EMPTY = 0-diff → the close-empty gate) and the unread per-file `patch` never stored. The route is greedy (`compare/*` — branch names carry slashes); the whole trimmed doc is stored per exact `(owner, repo, basehead)` in `compare_cache` — the row also splits `base_ref`/`head_ref` into their own columns (round 2), which is what enables the per-ref flush — and the compare's commits are also upserted into `git_commits_cache` (synergy; a hit never depends on those rows). Shape guard: NO query params are modeled (`per_page`/`page`/anything → passthrough), only the default JSON Accept (diff/patch media types → passthrough), a basehead with a colon (the cross-fork `owner:branch` form — its fork side can't be invalidated by this repo's pushes) or without `...` → passthrough. A 200 absorbs the comparison; a **404 absorbs as an unknown-ref VERDICT** (a status-404 row, `{"message":...,"status":"404"}` doc — the fleet's close-empty pass compares `base...head` for fork PRs whose head ref doesn't exist in the base repo, a 404 repeated every sweep by design); 5xx relays unstored. A comparison (or verdict) depends on both refs' tips/existence, so a `push` flushes every row naming the pushed ref on EITHER side (`base_ref`/`head_ref` match; ref creation/deletion arrives as a push, so a verdict clears the moment its missing ref is created), `repository` flushes repo-wide (covering renames), plus the 24h TTL backstop.

The blanket "no link fields" rule became a **pinned-allowlist mechanism** (the 2026-07-11 consumer survey superseded 2026-07-05's): the no-URL test takes an exact-key allowlist per route, and the per-status `target_url` + per-run `details_url`/`html_url` are pinned consumer-read exceptions (required-builds renders all three on its breakdown page) — adding another exception requires a fresh survey. Absorb gate: only a 200 whose body carries its array (statuses / check_runs / the bare array) absorbs — 404 (unknown ref) / 5xx relay unstored; an empty statuses array is cacheable. These routes deliberately do NOT read or write the `commit_checks` truth table: its normalized per-context rows are lossy against these responses (no timestamps, descriptions, or run ids); unifying the two representations is possible future work. Query-shape guard: only `state=open`, `per_page` (1..100), `page=1`, and `head=owner:branch` are modeled — anything else passes through. Pagination guard: a rebuilt list as long as the request's `per_page` is never served from state. Rebuilt item shape (trimmed, exactly a superset of what pr-minder + the pr-minder-reconcile hook read): `{node_id, number, state, title, user{login,type}, body, labels[{name,color}], created_at, updated_at, merge_commit_sha, draft, auto_merge (null | {merge_method}), head{ref,sha,repo:{full_name}|null}, base{ref,sha}}`.
     - `GET /repos/{owner}/{repo}/pulls/{number}` — the single-PR read (behind reveal), served from a `pull_requests` row ONLY when the row is **rest-complete** (`ghdata.PRRestComplete`: node_id + base sha present — GraphQL-sourced rows never are), **recently touched** (`ghdata.PRRowFresh`, the 24h staleness backstop for a missed `closed` delivery — `touched_at` is bumped by every webhook/absorb), AND its **`mergeable` is KNOWN** (MERGEABLE/CONFLICTING). `mergeable` is computed lazily by GitHub and pr-minder polls it to resolution; the known-mergeable gate is what makes caching safe — a null/unknown mergeable ALWAYS misses (fetch + absorb GitHub's computed answer, where a fetched null OVERWRITES via `SetPRMergeable` rather than COALESCE, so the poll keeps reaching GitHub until it resolves). A **branch push un-resolves** `mergeable` on affected open PRs (`NullPRMergeableByBranch`). A fetched CLOSED answer still DELETES any cached open row — the open-only invariant of `pull_requests` is untouched — but is now ABSORBED as a rendered whole-doc snapshot in `closed_pull_cache` and served from it thereafter (every pr-minder drain re-reads settled PRs; each such read used to be a fresh passthrough). The closed doc is the open rebuild's exact shape plus GitHub's real `merged` and its `mergeable` (key always present; typically null), `state` exactly `"closed"` — consumers short-circuit on `pr.state !== 'open'` and read nothing else from a closed PR. It is flushed per PR by EVERY `pull_request`/`pull_request_review` event (a reopen must not serve the stale doc; the REST open absorb also drops it), repo-wide by `repository`, bounded by a 24h TTL — and deliberately NOT flushed by `push` (a push cannot mutate a closed PR). A missed `reopened` delivery can therefore serve the stale closed doc for ≤24h — the same accepted staleness class as `PRRowFresh`, in the opposite direction. Any query string and non-numeric `{number}` segments pass through, as does any non-default `Accept` EXCEPT the diff read (next paragraph). Open rebuild = the list item shape + `{merged: false, mergeable: bool|null}`.

       **Merge-field invalidation semantics** (mergeable + merge_commit_sha): a push to a PR's **base or head ⇒ both fields are stale** — GitHub recomputes lazily and never webhooks the result, so `NullPRMergeableByBranch` nulls them AND **remembers the nulled test-merge sha** (`merge_stale_sha`/`merge_stale_at`). A tip change ALWAYS changes the test-merge sha, so **an absorb re-offering the remembered sha is stale by definition and can never re-resolve the row**: it stores (and the route serves) unresolved, the row keeps missing — each miss re-triggering GitHub's recompute — until a NEW sha arrives, which resolves and clears the marker (the guard lives in `UpsertPullRequest`'s SQL + `AbsorbSinglePull`, covering the single/list/webhook/sync writers uniformly; the marker rejects only within `ghdata.MergeStaleTTL` ≈ 1h == the strftime `'-1 hour'` cutoffs in the SQL, so a sha wrongly marked stale — absorbed post-recompute before a late push delivery — can't wedge the row into missing forever, and the 24h row TTL stays the outer backstop). Three closed holes (the webhooks#66 frozen-sha incident — a hit never reaches GitHub, so nothing triggered the recompute): the push handler once ran `SetRepoPushedAt` FIRST and returned on its error, skipping the invalidation (it now runs first, each step logs-and-continues); an unparseable push payload skipped it entirely (now: repo-wide `NullPRMergeableByRepo`, deliberately marker-less — the moved branch is unknown and an unmoved PR's re-offered sha is valid); and a recompute-lag refetch could re-resolve the just-invalidated sha permanently — while the sha-less GraphQL owner sync could re-arm `mergeable` via COALESCE on a null/stale-sha REST row (now gated; pure GraphQL rows keep sync-fed mergeable for the /graphql tier).

       The **diff read's 406 verdicts** (`respcache_pulldiff.go`): an Accept that is EXACTLY one media range naming the unified diff (`application/vnd.github.diff` or `.v3.diff`; multi-range Accepts keep the plain passthrough) gets its own flow — pr-minder's `getPullDiff` sends exactly that and branches ONLY on ok/406 (406 = "diff too large" → the files-API fallback; survey 2026-07-11), re-earning the same 406 on every describe hand-off for an oversized PR. The 406 VERDICT is cached per PR (`pull_diff406_cache`, doc `{"message": ...}`, served status 406) behind the SAME `pull` deny kind + resource key as the JSON read (same underlying resource, same authorization). A **200 diff BODY is deliberately never stored** — that would be verbatim byte caching, which the doctrine rejects (an opaque diff has no state to absorb) — so 200s and every other status relay unstored, every time. Flushed per PR by `pull_request`/`pull_request_review` (a head push can shrink the diff back under the boundary), repo-wide by `push`/`repository` (a BASE push can move any PR's three-dot diff across the boundary, with no per-PR signal), + 24h TTL.
     - `GET /repos/{owner}/{repo}/installation` — the App-level repo-installation lookup (pr-minder's `repoInstallationId`), registered OUTSIDE requireAuth exactly like the mint (App-JWT bearer, self-verified, app principal; `repo_installation_cache` table). Rebuild: `{id, account{login,type}, repository_selection, app_id, app_slug, target_type}`. Invalidated on `installation`/`installation_repositories` events by the stored installation id, plus a 24h TTL. 404 ("not installed") is never stored.

Whole-doc per-page snapshots in `pull_files_cache`, keyed `(owner, repo, number, per_page, page)`; trimmed per-file item `{filename, status, additions, deletions, changes, previous_filename?, patch?}` with the PRESENCE of `previous_filename` and `patch` preserved exactly (consumers test `typeof f.patch === 'string'`; GitHub legitimately omits `patch` on binary/oversized files) and the per-file blob `sha` + every URL field dropped. `patch` is unbounded, so a rendered document larger than **1 MiB passes through unstored** (the contents route's cap rule; the 8 MiB fetch cap still bounds the raw body). Shape guard: `per_page` (1..100, default 30) and `page` (1..**40**, default 1 — GitHub's files API stops at 3000 files = 30 pages at the consumers' `per_page=100`, plus margin for the trailing empty page and smaller per_page shapes; raised from 10 in round 2, deeper pages still pass through) only; unknown/repeated params, a non-numeric `{number}` (`/pulls/comments/...`), or a non-default Accept pass through. Only a 200 absorbs, all-or-nothing; an empty array (a page past the end) is a valid cacheable answer. Flushed per PR by EVERY `pull_request`/`pull_request_review` event (head pushes — including fork heads whose push events never arrive — base retargets, reopens), repo-wide by `push`/`repository` (the belt for missed PR deliveries), + 24h TTL.

     - `GET /repos/{owner}/{repo}` — the bare repository read (behind reveal; the reconcile hook's `getDefaultBranch`, plus status-only access checks in the buildhost `canAccessRepo` style, where the 200-vs-404 IS the answer). UNLIKE every other cached route there is NO snapshot table and NO per-row TTL: the rebuild comes straight from the `repos` TRUTH row — the same webhook-maintained, fleet-synced, consistency-checked state tier 1 serves — so truth freshness IS the freshness model (mirroring how `/graphql` serves truth; `repository` events correct rename/delete/visibility, and a reveal probe re-absorbs the row per principal within the ≤24h grant TTL, which bounds how stale a private repo's row can sit for a probing caller). Served only when the row can answer COMPLETELY: known visibility (unknown `''` fails CLOSED — e.g. a row seeded only by the identity-locked GraphQL fetch, which carries no visibility), a known `default_branch`, and a canonical full name; anything else fetches with the caller's token and the absorbed 200 heals the row. Trimmed rebuild: `{name, full_name, owner:{login}, private, visibility, default_branch, archived, disabled}` — `private` derived (visibility != public, so `internal` folds to private=true); `pushed_at` deliberately NOT emitted (truth `''` cannot distinguish never-pushed from not-yet-synced, and no consumer reads it); `fork`/`id` not stored, so not emitted. Query params, non-default Accepts, and HEAD requests (MethodNotAllowed → the proxy) pass through; a non-200 fetch relays unstored.
     Truth rows are global; the reveal layer is what keeps callers apart. Invalidation deletes are global too (a webhook is a global fact). Every response-cache table is pruned on write: expired rows plus LRU beyond `ghdata.CacheMaxRows` (default 1,000,000; set at startup from the `CACHE_MAX_ROWS` env var — unparseable/<1 fails startup — and lowered directly by tests). One knob for every table on purpose: all but `git_commits_cache` are TTL-bounded, so the cap is only a runaway safety net for them; `git_commits_cache` (immutable rows, no TTL) is the one table that actually grows to the ceiling. Dispositions land in the request log: `hit` (served from state), `miss` (fetched+absorbed, with upstream status), `passthrough` (read forwarded/replayed unstored), `write` (mutation proxied), `error`.
  3. **Everything else — verbatim passthrough, uncached.** Transparently reverse-proxied to GitHub via `newGitHubProxy` (`internal/api/proxy.go`), wired as chi's `NotFound`/`MethodNotAllowed` (so the `Use` middleware — CORS, recoverer — still wraps it); the GraphQL handler forwards any non-org query the same way. The proxy forwards the caller's `Authorization` verbatim (never the server token), rejects tokenless requests with 401, targets `gh.BaseURL()` (tests hit the fake upstream), and `ModifyResponse` strips GitHub's duplicate `Access-Control-Allow-*` while keeping `Access-Control-Expose-Headers`. A 2xx passthrough under `/repos/{owner}/{repo}/...` refreshes the caller's grant (GitHub just proved access — `refreshGrantOn2xx`). Mutating methods are logged as `write`, not `passthrough` — a write is proxied because it mutates, not because caching failed. Do NOT make this path touch the freshness store.

  Adding a cached REST route means tier 2: reveal gate + global structured state + trimmed rebuild + the recursive no-URL-keys invariant test + explicit invalidation/TTL/pruning — never verbatim byte replay, and never a byte-identity requirement (that is tier 1's, only).
- **Subscriber notifications (`internal/notify`) — config DB, reveal-gated fan-out, drained delivery** — the rules that must not regress:
  1. **Subscriptions are CONFIG in their own SQLite file, never nuked.** The SchemaVersion bump-to-nuke doctrine applies ONLY to the main cache DB; the subscriptions DB (`SUBSCRIPTIONS_DB_PATH`, default `DeriveDBPath(DB_PATH)` = strip trailing `.db` + `-subscriptions.db`) is opened with `CREATE TABLE IF NOT EXISTS` and may only ever evolve ADDITIVELY (new columns with defaults) so old files keep working under new binaries. Its queries are **hand-rolled SQL inside `internal/notify`** — a deliberate, documented exception to the sqlc doctrine: `sqlc.yaml` targets the main cache DB's schema, and a second codegen target for one small config table is not worth the entanglement.
  2. **Reveal-gated, fail-closed fan-out.** A subscription is notified about a repo ONLY if its principal could read it via the reveal layer's NON-PROBING fast paths at delivery time: public visibility in global truth, or a live grant (`AccessChecker` = `GetRepoInsensitive` + `HasGrant`, satisfied by `ghdata.Store`). There is NO per-notification probe (no caller token exists at delivery time) and no deny-cache involvement; unknown/absent visibility = private, and a gate-store error gates the notification. A private repo's activity must NEVER reach an unauthorized principal.
  3. **Detached, tracked, drained delivery.** Notifier work runs on `context.Background`-derived contexts with bounded timeouts (the freshness-fetch pattern), tracked in a WaitGroup behind a mutex-guarded `track()` (closes the Add-vs-Wait race); `Notifier.Drain(timeout)` stops new work, cuts pending backoff sleeps, and waits out in-flight deliveries — `cmd/server` drains it alongside the freshness manager BEFORE closing the DBs, and closes the subscriptions DB after the main one. `NotifyIngest` returns immediately (the GitHub webhook response never waits on subscriber POSTs); only `applied`/`invalidated` dispositions and single-owner/repo deliveries notify (repo-less `installation` events are skipped).
  4. **Bounded retries + auto-disable.** ~8 concurrent POSTs, 3 attempts each (~8s per attempt, ~1s/~4s backoff); a terminal failure bumps `consecutive_failures` (short `last_error`, never the secret); at 10 the subscription auto-disables (`active=0`, `disabled_reason` set, WARN log). Success resets the counter. `PATCH {"active": true}` is the re-enable path (resets counter, clears reason).
  5. **Secrets at rest.** Subscription secrets are stored plaintext to sign every delivery (`X-Hub-Signature-256` over the exact body, per attempt) — the same trust domain as `install_token_cache` tokens. They are never logged and never returned by any API response (`subscriptionJSON` structurally has no secret field).
  6. **`/_mirror/*` is the RESERVED mirror-native namespace** — GitHub has no underscore-prefixed top-level paths and chi routes beat the NotFound passthrough, so it can never collide with proxied GitHub traffic; new mirror-native endpoints belong under it. The CRUD routes live INSIDE requireAuth (per-principal ownership; a foreign id answers 404 — no existence leak), skip the reveal layer (not repo reads), and stay out of the GitHub request log. Observability is the admin-only `GET /api/notifications` (in-memory ring + counters, requestLog stance) — JSON only, no dashboard tab, no TS.
- **Installation tokens at rest (security note)** — the token-mint cache stores each minted installation token in SQLite (`install_token_cache.token`). This is the same trust domain as the traffic itself: the mirror already relays these exact tokens in cleartext on every request it proxies, the DB file sits on the same host with the same access, and every row is bounded by the token's own short (~1h) expiry plus the serve-until cutoff. Rows are pruned on write (expired + LRU) and flushed on installation events. Token values are never logged (the request log records method/path/disposition only) and never appear in error messages; keep it that way when touching `respcache.go`.
- **OAuth relay (github.com login endpoints)** — `POST /login/oauth/access_token` and `POST /login/device/code` (`internal/api/oauth.go`, one shared `relayGitHubLogin` core) relay the two browser-blocked `github.com` login endpoints (NOT `api.github.com`): the OAuth code-for-token exchange and the RFC 8628 device-authorization start, each returned with the mirror's CORS headers so a purely client-side app can complete login (GitHub's login endpoints send no CORS headers). The device flow's polling leg (`grant_type` `urn:ietf:params:oauth:grant-type:device_code`) goes through the access_token route unchanged — the body is opaque bytes to the relay. Both are registered outside `requireAuth` (the body is the credential — an OAuth `client_secret`, or just a public `client_id` for the device flow; no bearer token) and forward only the body + content-negotiation headers to fixed URLs (`githubOAuthTokenURL`/`githubDeviceCodeURL`, vars so tests can point them at a fake). They do NOT copy GitHub's `Access-Control-*` (corsMiddleware is the single CORS authority, so no duplicate ACAO). This is why repo-nightmare's `TOKEN_PROXY` points here instead of a standalone CORS proxy.
- **Dashboard = separate authz model** — the web dashboard (`internal/api/dashboard.go`, served at `/`) authenticates a human via **GitHub OAuth** and authorizes by **login** (session cookie), which is deliberately distinct from the data API's bearer-token + reveal model. It reports the GLOBAL truth totals (one cache, one truth — same numbers for everyone) plus per-**principal** reveal-layer standing: live grant count, org-sync freshness, recent refreshes. A normal user sees their own principal(s); logins in `ADMIN_LOGINS` (default `PazerOP`) see every known principal. The `actor_identities` table maps principal→display name (a user's login, an app's slug, or an installation's account login) for this labeling only — written by `requireAuth`, the self-verifying app-JWT routes (token mint, repo installation), and the background refresher's installations listing; names come only from verification calls that already happen, never extra GitHub calls, and every surface falls back to the bare key. Admins additionally get: the **webhook delivery log** (`GET /api/webhooks`, "Webhooks" tab — dispositions are `applied`/`invalidated`/`ignored`/`error`; there is no `skipped`), the **request-activity log** (`GET /api/requests`, in-memory `requestLog`, "Requests" tab — `hit`/`miss`/`passthrough`/`write`/`error` with per-disposition totals, plus **route-shape groups**: `requestgroups.go`'s `normalizeRoute` folds each path onto its shape (`/repos/{owner}/{repo}/compare/{basehead}`, ...) and the log keeps bounded cumulative per-group disposition counters — since-restart like the totals, NOT windowed by the recent ring — which the tab renders as two tables above the flat history, "Top requests" and "Top uncached requests" (passthrough > 0, the caching candidates); the payload also carries the SQLite DB's on-disk size — `db_size_bytes`/`db_wal_size_bytes`, `DB_PATH` + its `-wal` sidecar statted per request, omitted when missing — shown on the tab's summary line), the **rate-limit view** (`GET /api/ratelimit`, "Rate limit" tab — `{live, observed, note?}`: `live` is the GitHub App's per-installation `GET /rate_limit` poll, `observed` is the `internal/ratemeter` snapshot of passively recorded `X-RateLimit-*` headers per (identity, resource); with no App configured — or a failed poll — the endpoint answers 200 with an empty `live`, a `note`, and the observed data, never a bare 503), the **Timeline chart** (`GET /api/timeline?since=<id>`, "Timeline" tab — the `internal/reqtimeline` ring rendered as a swimlane chart: one lane per webhook event type / per proxied route shape, real measured durations, ms-scale webhooks as native instant pips; the tab's `<gsm-timeline>` element is created ONCE and kept alive across refreshes — it polls with the `since` cursor and `mergeData`s in place, deliberately breaking the other tabs' wipe-and-rebuild convention because the canvas holds viewport state; the `<timeline-view>` component itself is imported at RUNTIME from js-snippets' GitHub Pages, never vendored — fix component bugs upstream — with a forever-retrying, cache-busted dynamic import and a "chart loading…" note while unreachable), and `GET /api/jobs` (workflow jobs; no dashboard tab yet). The request log, the rate meter, and the timeline ring are **in-memory** (live views, not audit logs) so they reset on restart.
- **Admin truth browse, grants view + consistency check/reconcile (admin-only operator tooling)** — `GET /api/cache/data` dumps the GLOBAL truth rows (repos with visibility, open PRs, commit checks) as flattened JSON; `GET /api/cache/data?principal=<key>` instead dumps that principal's **access grants** (who can see what, each grant's source and expiry) — surfaced as the "Browse truth" / "Run consistency check" / "Reconcile" buttons on the admin Principals tab and a per-principal "Grants" button. `GET /api/cache/check[?org=<owner>]` runs the **consistency check** (`internal/sync/consistency.go` + `consistency_diff.go`): it re-fetches the source of truth from GitHub via the mirror's **GitHub App** (`AppAuthenticator`, shared with background refresh) using the owner-agnostic `GetOwnerData` + `OwnerRepoVisibilities` — so **User-account installations are checked like Organizations** (only owners with no installation are skipped, under `orgs_skipped`) — and diffs GLOBAL truth against it. Compared: repo/PR existence both ways; repo fields `default_branch_status`, `default_branch`, `is_disabled`, `url`, `is_archived`, `pushed_at` (5-minute tolerance — lag = missed push webhooks = possibly stale `contents_cache`), and `visibility` — the reveal layer's security field: cached-public/GitHub-nonpublic gets its own `visibility_leak` issue + summary tally, cached `''` reports as informational `visibility_unknown`; PR fields incl. `last_commit_status` and `auto_merge_method` (fetched via `autoMergeRequest`), plus labels (100 per PR on this path). `mergeable` is deliberately not compared — the cache un-resolves it on pushes. An only-in-cache repo that is **archived** (the visibility twin has no `isArchived` filter) is classified expected, not drift (tallied in `repos_only_in_cache_archived`); cached-open PRs under an only-in-cache repo are swept as their own `pr only_in_cache` entries; PR-existence entries carry `served_now` (derived from the repo's live `pulls_list_cache` marker, read WITHOUT touching its LRU) and only-in-cache PR entries carry the cached title/updated_at/touched_at; every discrepancy gets a per-class `fix` hint; a failed visibility fetch becomes a per-owner report note, never a silent gap. The report carries `truth_freshness`: per owner, the NEWEST org list-sync marker any principal holds. The GET is **read-only** and compares **OPEN** PRs only; returns **503** when no App is configured. **Streaming mode** (`?stream=1`, GET and POST alike): the same route answers `application/x-ndjson` — one JSON line per checker progress event (`internal/sync/progress.go` `ProgressEvent`, phases start/owner/fetch/visibility/diffed/applied/skip/done; the per-page fetch events come from `ghclient.GetOwnerDataWithProgress` + the owner query's `totalCount`, an owner-only extra the locked org query never grows), each line flushed immediately (Cloudflare-safe), ending with `{"phase":"report",report}` carrying the exact non-stream report (or `{"phase":"error"}`); the run stays on the request context so closing the dashboard modal aborts it (apply corrections are idempotent), non-stream responses are unchanged, and the dashboard modal renders these lines as a live per-owner/per-page progress bar (`internal/api/checkstream.go`). **Apply mode** (`POST /api/cache/check?apply=true`, alias `?apply=1`; `?apply` on a GET is a 405 — `internal/sync/consistency_apply.go`): after the diff, the drift is CORRECTED from the same fetched snapshot, per owner under `AppInstallationActor(inst.ID)` with `fetchStart` captured before the fetch — (1) `SyncOrgTruth` absorbs repos + open PRs + labels + statuses and deletes stale cached-open PRs (grace-guarded); (2) `SetRepoVisibility` from the visibility map (fail-closed `''` rows and drift both directions); (3) `SetRepoDefaultBranchStatus` with GitHub's value INCLUDING NULL (never rely on the COALESCE upsert); (4) the **commit_checks stick rule**: a terminal GitHub rollup (SUCCESS/FAILURE/ERROR) contradicted by recorded rows → `ForceCheckRollup` (one tx: delete the sha's rows + `SetPRStatusByHeadSha`); an empty GitHub rollup with leftover rows or a non-NULL column → the same with NULL; PENDING/EXPECTED → no-op; per-check rows are NEVER synthesized (a fabricated completed row would be tomorrow's ghost), and the correction STICKS because the next PR webhook's `UpsertPRWithChecks` derives an empty rollup (guard skips) while COALESCE preserves the set value; (5) `SetPRAutoMergeMethod` including NULL (a stale armed flag would silently disable pr-minder's merge backstop); (6) the app-installation freshness marker is stamped. The response is the normal report plus an `applied` tally (repos/PRs absorbed, PRs deleted, visibility/statuses/default-branch/auto-merge set, check rows deleted); discrepancies always show the PRE-apply state. Known accepted race: `commit_checks` rows carry no timestamp, so a webhook landing between the fetch and the correction can lose one write (it re-appears on that check's next event).
- **TypeScript front-end** — the dashboard JS is authored in `internal/api/web/src/*.ts` and compiled by `npm run build` (tsc) to `internal/api/web/assets/*.js`, which is **committed as a generated artifact** and embedded via `//go:embed`. Edit the `.ts`, never the `.js`. CI's `web-check` job wipes the generated `.js` (hand-written `assets/style.css` survives), recompiles with the locked toolchain (`npm ci`), and fails on any divergence — stale committed JS, an emitted file never committed, or an orphaned `.js` left by a deleted/renamed `.ts` (run `npm run build` and commit). `assets/demo-data.js` is preview-only (NOT embedded); the CI `preview` job injects it to deploy a backend-free styling preview to buildhost per branch. Each `src/*.ts` emits its own standalone ES module loaded by its own `<script type="module">` tag in `index.html` — `rate-meter.ts` self-registers the `<rate-meter>` web component; `timeline.ts` self-registers `<gsm-timeline>` (types for its Pages-URL component import live in the interim ambient shim `src/js-snippets-timeline.d.ts` — types only, keep it minimal). Adding an asset file means wiring it in **three** places: the `//go:embed` + hashed-URL machinery in `internal/api/dashboard.go`, the `index.html` script tag, and the `preview` job's copied-assets list in `ci.yml`.
- **Build** — use `go-toolchain` (not `go build`/`go test` directly); run `npm run build` after editing `web/src/*.ts`

## Environment Variables

- `GITHUB_APP_ID` (optional) — GitHub App ID. When set with a private key, the service signs in as this app for background work — periodic refreshes and the source-of-truth fetch for the admin consistency check (`/api/cache/check`); it is never used to serve normal data-API requests, which require the caller's own `Authorization` header (401 otherwise). Unset = periodic refreshes disabled and the consistency check reports "unavailable" (503). Webhooks apply to global truth either way.
- `GITHUB_APP_PRIVATE_KEY` / `GITHUB_APP_PRIVATE_KEY_PATH` — the App's PEM private key (PKCS#1 or PKCS#8), inline (optionally `\n`-escaped) or as a file path (path wins). Required when `GITHUB_APP_ID` is set.
- `WEBHOOK_SECRET` — GitHub webhook HMAC secret
- `LISTEN_ADDR` — HTTP listen address (default `:8080`)
- `DB_PATH` — SQLite database file path (default `github-mirror.db`)
- `SUBSCRIPTIONS_DB_PATH` (optional) — the subscriber-notification config DB, a SEPARATE SQLite file the SchemaVersion nuke never touches. Default derives from `DB_PATH`: strip a trailing `.db`, append `-subscriptions.db` (`github-mirror.db` -> `github-mirror-subscriptions.db`).
- `ALLOWED_ORIGINS` (optional) — comma-separated CORS allow-list for browser clients (e.g. the repo-nightmare PR viewer). Defaults to `*` (any origin), which is safe because the cache reveals data per the caller's proven GitHub access, not origin. Preflight `OPTIONS` is answered without auth; see `internal/api/cors.go`.
- `CACHE_MAX_ROWS` (optional) — per-table row ceiling for the response caches (sets `ghdata.CacheMaxRows` at startup; default `1000000`). One knob for every cache table: all but `git_commits_cache` are TTL-bounded, so the cap is only a runaway safety net for them; `git_commits_cache` (immutable rows, no TTL) is the one table that actually grows to the ceiling. A value that is unparseable or < 1 fails startup.
- `GITHUB_OAUTH_CLIENT_ID` / `GITHUB_OAUTH_CLIENT_SECRET` (optional) — GitHub OAuth App credentials for dashboard login. If unset, the dashboard still renders but sign-in is disabled. Register the OAuth App's callback as `<BASE_URL>/auth/callback`.
- `SESSION_SECRET` (optional) — HMAC key for session cookies. If unset, a random per-process key is used (sessions reset on restart); set it in production.
- `ADMIN_LOGINS` (optional) — comma-separated logins that get the admin dashboard views (all principals, truth browse, consistency check; default `PazerOP`); case-insensitive.
- `BASE_URL` (optional) — public base URL (e.g. `https://github-state-mirror.pazer.io`) used to build the OAuth `redirect_uri`; derived from the request (honoring `X-Forwarded-Proto`) when unset.
