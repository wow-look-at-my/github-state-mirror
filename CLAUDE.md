# GitHub State Mirror

## Project Overview

A Go service that mirrors GitHub state into SQLite, providing a fast local API surface. This is a **cache** — the SQLite DB is disposable and gets nuked+recreated on schema version changes.

## Architecture

- `internal/actor/` — Context-based principal-key propagation + display shortening (stdlib-only, safe to import from anywhere). A "principal" identifies WHO is asking (`user:<id>`, `app:<id>`, `app-installation:<id>`, or a token fingerprint) — it gates what the cache REVEALS, never what it STORES.
- `internal/freshness/` — Generic cache freshness framework (zero GitHub knowledge); `Manager.Drain` waits out in-flight detached fetches at shutdown
- `internal/database/` — SQLite schema + sqlc-generated queries (`dbgen/` is codegen, do not edit)
- `internal/ghdata/` — Domain store wrapping sqlc with transaction logic: global truth writes (`SyncOrgTruth`, `UpsertPRWithChecks`, ...), the reveal layer's `access_grants`/`deny_cache`, and the dashboard reads. The grant-writing paths (`SyncOrgTruth`, `RecordGrant`) opportunistically sweep expired grants/denials after committing (`maybePruneAccessControl`: best-effort, log-on-error, throttled to once per ~10 min via an atomic last-run stamp); reads (`HasGrant`, `GrantsByPrincipal`, `CountLiveGrants`, `ListGrantPrincipals`) all filter on expiry regardless
- `internal/ghclient/` — GitHub REST + GraphQL API client (includes the in-memory token→identity cache behind `ResolveTokenIdentity` and `AppAuthenticator` for GitHub App JWT / installation-token minting). `SetRateObserver` installs the passive rate-limit hook: every response the client sees is reported to `internal/ratemeter`, labeled by the ctx principal when set, else by the credential's shape (`app-jwt` for a JWT, a short `token:<fp12>` fingerprint otherwise — never the raw token)
- `internal/ratemeter/` — Passive rate-limit observation: a thread-safe, IN-MEMORY store of the latest `X-RateLimit-*` reading per (identity, resource), fed by every upstream path (passthrough proxy, cached-route miss fetches, reveal probes, ghclient calls) and read by the dashboard's "Rate limit" tab. Deliberately in-memory (the `requestLog` stance: live view, not an audit log — and a DB table would force a cache-nuking SchemaVersion bump for sub-hour-ephemeral data); bounded at 512 entries (oldest-ObservedAt evicted); methods are nil-receiver-safe; resets on restart
- `internal/auth/` — GitHub OAuth login + signed-cookie sessions for the dashboard (stdlib-only, no DB knowledge)
- `internal/sync/` — Bridge layer: the org list-sync fetcher, periodic refresh, webhook dispatch (always-apply), the global consistency checker
- `internal/webhook/` — Webhook HTTP handler + event parsing (dispatch is **synchronous**; the handler returns a per-delivery disposition in the response so GitHub's delivery record reflects whether the cache was updated)
- `internal/api/` — chi router, REST handlers, GraphQL cache assembler, the **reveal layer** (`reveal.go`), the GitHub passthrough proxy (`proxy.go`), the in-memory request-disposition log (`requestlog.go`), and the web dashboard (`dashboard.go` + embedded `web/`)
- `internal/api/web/` — Dashboard front-end: `src/*.ts` (TypeScript source) compiled to `assets/*.js` (committed, generated, embedded)
- `cmd/server/` — Entry point (drains in-flight fetches before closing the DB on shutdown)

## The global-truth model (operator directive, 2026-07-03)

> "just because nobody has fetched something doesn't mean we get to ignore updates from webhooks for it. there's only 1 cache because there's only 1 true state, the appropriate parts of the cache are revealed to users/apps based on their permission model (i.e. were they able to access this resource directly through github -> yes -> ok we can serve the cache for this resource to them)"

Storage and authorization are **separate axes**:

1. **ONE global truth store.** `repos`, `pull_requests`, `pr_labels`, `commit_checks`, `contents_cache`, `git_commits_cache`, `pulls_list_cache` carry NO actor column — there is one row per fact because there is one true state. Webhooks and every principal's fetches all maintain the same rows. (`install_token_cache` and `repo_installation_cache` stay keyed by the app principal: a minted token IS per-credential, not shared truth.)
2. **Webhooks ALWAYS apply.** The dispatcher never asks "who has this cached?" — a delivery for a repo nobody ever fetched upserts the `repos` row straight from the payload's own `repository` object (`absorbRepoFromPayload`) and applies. There is no `skipped` disposition, no on-demand pull, and no `ActorsForRepo` gate; those were deleted.
3. **Reads reveal by permission** (`internal/api/reveal.go`). Serving a cached resource requires proof the caller could read it from GitHub directly: the repo is **public** in truth (fast path, no GitHub call), the principal holds a live **access grant**, or a live **probe** (`GET /repos/{owner}/{repo}` with the caller's own token) answers 2xx. Grants are earned only from GitHub's own answers — an org list-sync **replace-syncs** the principal's grants to exactly the repos GitHub returned them (`source=list_sync`), and a successful probe records one (`source=probe`); TTL ~24h (`ghdata.GrantTTL`). Authoritative denials (404, non-rate-limit 403) are cached in `deny_cache` for ~5m (`ghdata.DenyTTL`) and replayed without touching GitHub; transient failures (5xx, 429, rate-limited 403, network) are NEVER cached and fail only that one request. Unknown visibility (`repos.visibility` = `''`) is treated as private — fail closed.

## Key Constraints

- **No migrations** — bump `SchemaVersion` in `internal/database/db.go` to nuke+recreate
- **sqlc codegen** — run `sqlc generate` after modifying `schema.sql` or `queries/*.sql`
- **Freshness/data separation** — `internal/freshness/` must never import GitHub-specific packages
- **Lazy fetches are detached, backoff-gated, and drained** — `freshness.Manager.doFetch` runs the fetch and its metadata writes on a `context.WithoutCancel`-derived context bounded only by `fetchSafetyTimeout` (5m): a client aborting its request cannot kill the shared multi-page org-repos fetch mid-flight or prevent the result from being stored, and a short caller deadline is deliberately NOT re-applied (the old webhook-deadline re-application killed fetches half-done). Every detached fetch is tracked in an inflight WaitGroup; `Manager.Drain(timeout)` waits for them and `cmd/server` calls it after `ListenAndServe` returns, BEFORE `db.Close()` — so a fetch can never write to a closed DB at shutdown. An error-state row inside its `retry_after` window is NOT re-fetched by lazy callers — `EnsureFreshOutcome` reports the stored error instead, so a failing upstream isn't hammered on every request; deliberate refreshes (periodic triggers) bypass the backoff. When `/graphql` serves the stale cache after a refresh failure it sets `X-GSM-Stale: true` + `X-GSM-Last-Fetched` response headers and WARN-logs with the principal — headers only, the body stays identity-test-identical.
- **One global truth, revealed by permission (replaces per-actor isolation)** — see "The global-truth model" above; this is the load-bearing security section. The principal is resolved in `requireAuth` (`internal/api/router.go`): a user token resolves via `ghclient.ResolveTokenIdentity` (`GET /user`, cached per token) to `user:<numeric id>` (stable across renames, shared by all of that user's tokens); a token that is definitively NOT a user (non-rate-limit 403 / 404 on `/user`) falls back to the SHA-256 token fingerprint; a transient `/user` failure with no cached verdict fails 503 (never a silent guess). The security boundary is **the reveal layer, not storage**: one user can never read another's private-repo data because revealing `owner/repo` to a principal requires the repo to be public, a live grant (earned from GitHub answering THAT principal), or a live probe with THAT principal's own token. Public repos are served to any authenticated caller. Fail-closed rules that must not regress: unknown visibility = private; deny verdicts cache only authoritative answers (a rate-limited 403 must never pin a legitimate caller out for the TTL); grants expire (~24h) so revoked GitHub access converges; `SyncOrgTruth` REPLACE-syncs list_sync grants (a repo GitHub stopped returning loses its grant). Data endpoints reject tokenless requests with 401. There is **no static service token**: the **GitHub App** (`ghclient.AppAuthenticator`) is the service's only credential, used for background work (periodic refreshes and the consistency check), never to serve requests; the periodic refresher mints a short-lived access token per installation and runs as the stable `app-installation:<id>` principal (see `sync.AppSessions`), so its syncs both refresh global truth and hold that principal's grants.
- **Webhooks always apply (the whole point)** — the dispatcher (`internal/sync/webhook.go`) applies payloads **directly** to global truth so high-frequency events never trigger a GitHub re-fetch: `pull_request`/`pull_request_review` upsert the PR (and upsert the repos row from the payload first, so a never-fetched repo is absorbed on its first delivery); `status`/`check_run`/`check_suite` aggregate per-check state in the `commit_checks` table and roll it up onto each PR's `last_commit_status` (and the repo's `default_branch_status` when the check ran on the default branch); `push` updates `pushed_at`, absorbs the pushed commits into `git_commits_cache`, and **un-resolves `mergeable`** on every open PR whose base or head is the pushed branch (`NullPRMergeableByBranch` — GitHub recomputes mergeability after either side moves and never webhooks the result); `label` recolors/removes labels; `repository` handles lifecycle (deleted → `DeleteRepoCascade`, renamed → delete the old name + absorb the new, privatized/publicized → `SetRepoVisibility`, other actions → absorb or invalidate). Invalidation (`MarkStaleByKindKey`) is only a **fallback** for structural events with unparseable payloads. Do NOT regress this into invalidate-and-refetch, and do NOT add a "who has this cached?" gate — truth is maintained unconditionally. `UpsertPullRequest` COALESCEs `last_commit_status` AND `mergeable` against the existing row, so a PR webhook — which carries no CI state, and carries `mergeable: null` while GitHub is still computing it — doesn't clobber known values (the un-resolve signal is the PUSH event, not the PR event's transient null); a genuinely resolved `mergeable` still overwrites. GraphQL-sourced REST-only columns (`node_id`, `body`, `auto_merge_method`, `merge_commit_sha`, ...) are guarded the same way so a GraphQL sync can't blank what a webhook/REST absorb recorded. `UpsertPRWithChecks` additionally derives `last_commit_status` from the commit checks already recorded for the PR's head sha, so a PR opened *after* its head commit's CI finished (pr-minder auto-opened PRs) doesn't sit at NULL forever. The handler dispatches **synchronously** and returns a `DispatchResult` whose disposition — `applied` (200) / `ignored`·`invalidated` (202) / `error` (500) — is the response GitHub records (there is NO `skipped`); every delivery is also written to the global `webhook_deliveries` log. Don't restore the old fire-and-forget 200. The dispatcher also maintains the response caches (disposition-neutral, best-effort): `push`/`repository` flush the repo's `contents_cache` rows, `repository` additionally flushes the repo's `pulls_list_cache` marker (`pull_request` events deliberately do NOT — they maintain the PR rows the marker vouches for), `installation`/`installation_repositories` flush the installation's `install_token_cache` AND `repo_installation_cache` rows. `workflow_job` deliveries (`in_progress`/`completed` only; `queued`/`waiting` dropped as `ignored`) record job state in the global `workflow_jobs` table (admin-only `GET /api/jobs`; out-of-order tolerant upsert; completed rows pruned after 14 days). **Operator action required**: the GitHub App must be subscribed to the `workflow_job` event for that table to fill.
- **App-identity principal (opt-in, for trusted first-party app callers)** — a *request*-time caller may send a **GitHub App JWT** in the `X-Mirror-Identity` header. `requireAuth` verifies it via `ghclient.VerifyAppIdentity` (`GET /app` — GitHub only 200s if the RS256 signature checks out against the app's public key, so it's unforgeable; cached per-JWT) and resolves that caller to the principal `app:<id>`, skipping the per-user/fingerprint resolution entirely (installation tokens cannot call `/user`). This exists because such a caller's installation tokens rotate hourly — fingerprint principals would lose their earned grants every hour; the app principal keeps **one stable grant set** across all the app's tokens. The `Authorization` token is still injected into the context for upstream fetches/probes/passthrough, so per-repo authorization against GitHub is unchanged. No identity header → the default per-user/fingerprint resolution above.
- **The cache contract (three tiers)** — every data route falls into exactly one of these; know which tier a route is in before touching it:
  1. **GraphQL org-repos — byte-identical, identity-test-locked (unchanged).** `POST /graphql`'s org-repos query is assembled in `graphql.go` to match exactly GitHub's selection set, locked by an identity test (`identity_shape_test.go`). This is the ONLY route where byte-identity to GitHub applies. The assembly reads global truth **filtered by the reveal layer** (`ListVisibleReposByOwner`: public repos + the principal's granted repos), so the served view tracks what GitHub itself answers this principal — never the whole truth store. The backing fetch (`OrgReposFetcher`) runs with the PRINCIPAL'S OWN token, merges the snapshot into global truth (`SyncOrgTruth`: upsert + reconcile, never delete-then-insert), and replace-syncs the principal's grants. Any other GraphQL query forwards to the passthrough (a `mutation` is recorded as a `write`).
  2. **Cached REST routes — global state absorbed, rebuilt TRIMMED, revealed by permission.** `internal/api/respcache.go` + `respcache_pulls.go` + `respcache_installation.go` + `internal/ghdata/respcache*.go`. Every read passes the reveal gate first (`reveal(r, owner, repo, kind, key)`): public repo / live grant / cached deny verdict / live probe — a denied caller gets GitHub's own relayed answer, an unrevealable transient error 502s that one request, and a successful probe absorbs the probed repo object into truth and records a grant. The route ABSORBS the state a GitHub response contains into structured columns and REBUILDS the response from that state, **dropping every URL field**: `url`, anything matching `*_url`, and `_links` (locked by `TestRespCache`/`assertNoURLKeys` in `respcache_test.go`). Hits AND misses both serve the rebuilt shape, marked `X-GSM-Cache: hit|miss`; rebuilt responses carry only `Content-Type: application/json; charset=utf-8`. Anything a route cannot model passes through verbatim, unstored, recorded as `passthrough`: non-default `Accept` media types, unknown query params, symlink/submodule objects, >1 MiB `"encoding":"none"` files, oversized bodies, unexpected statuses. The v1 routes:
     - `GET /repos/{owner}/{repo}/contents/{path...}` (incl. `?ref=`, part of the key) — absorbs 200 file (base64), 200 dir (trimmed entries), **and 404** (the "config file absent" answer is half the win). One global row per (owner, repo, path, ref). Invalidated on `push`/`repository` webhooks (whole-repo flush) plus a **24h TTL backstop**.
     - `GET /repos/{owner}/{repo}/git/commits/{sha}` — immutable: no invalidation, no TTL, LRU-bounded only. Rebuild: `{sha, author{...}, committer{...}, message, tree:{sha}, parents:[{sha},...]}`. **Push payloads feed this cache too** (`absorbPushCommits`): each pushed commit is upserted once, globally; parents come from the payload's linear chain (`webhook.PushPayload.ChainedCommits`), which declines forced pushes, new refs, possibly-truncated arrays (≥20), and non-linear orderings rather than derive wrong parents. 404s are never cached (a missing sha can be pushed later).
     - `POST /app/installations/{id}/access_tokens` — the token-mint cache, registered OUTSIDE requireAuth (an App JWT cannot resolve `GET /user`): the handler verifies the bearer itself via `ghclient.VerifyAppIdentity` and keys by the app principal. Key: (app id, installation id, SHA-256 of the **canonicalized** request body). Only a 201 whose `expires_at` parses is stored, served until `expires_at - 10min`; rebuild is exactly `{token, expires_at, permissions?, repository_selection?}`. Invalidated on `installation`/`installation_repositories` events. An unverifiable bearer forwards unchanged, uncached. NOT global truth — a minted token is per-credential.
     - `GET /repos/{owner}/{repo}/pulls` — the open-PR list (behind reveal), absorbed into (and rebuilt from) the SAME global `pull_requests`/`pr_labels` tables the webhooks maintain — no separate state table. Serving a LIST from rows needs a completeness proof, so a per-repo GLOBAL **"open-PR list complete" marker** (`pulls_list_cache`) gates it: set when an *unfiltered page-1* response absorbs with fewer items than `per_page` (which also reconciles cached open rows the response no longer contains, guarded by the fetch-start grace so a webhook racing the fetch wins); **cleared by `repository` events**; **never touched by `pull_request` events** — those maintain the rows, which is the point (a 24h TTL backstop bounds missed deliveries). A GraphQL sync can add rows that LACK the REST-only columns; the rebuild-time `allRestComplete` gate turns that into a miss rather than serving a hole, so the marker needs no clearing there. Query-shape guard: only `state=open`, `per_page` (1..100), `page=1`, and `head=owner:branch` are modeled — anything else passes through. Pagination guard: a rebuilt list as long as the request's `per_page` is never served from state. Rebuilt item shape (trimmed, exactly a superset of what pr-minder + the pr-minder-reconcile hook read): `{node_id, number, state, title, user{login,type}, body, labels[{name,color}], created_at, updated_at, merge_commit_sha, draft, auto_merge (null | {merge_method}), head{ref,sha,repo:{full_name}|null}, base{ref,sha}}`.
     - `GET /repos/{owner}/{repo}/pulls/{number}` — the single-PR read (behind reveal), served from a `pull_requests` row ONLY when the row is **rest-complete** (`ghdata.PRRestComplete`: node_id + base sha present — GraphQL-sourced rows never are), **recently touched** (`ghdata.PRRowFresh`, the 24h staleness backstop for a missed `closed` delivery — `touched_at` is bumped by every webhook/absorb), AND its **`mergeable` is KNOWN** (MERGEABLE/CONFLICTING). `mergeable` is computed lazily by GitHub and pr-minder polls it to resolution; the known-mergeable gate is what makes caching safe — a null/unknown mergeable ALWAYS misses (fetch + absorb GitHub's computed answer, where a fetched null OVERWRITES via `SetPRMergeable` rather than COALESCE, so the poll keeps reaching GitHub until it resolves). A **branch push un-resolves** `mergeable` on affected open PRs (`NullPRMergeableByBranch`). Closed/unknown PRs pass through — absorbing a closed answer DELETES the cached row (the cache retains open PRs only). Non-default `Accept` (the `application/vnd.github.diff` read) and any query string pass through; so do non-numeric `{number}` segments. Single rebuild = the list item shape + `{merged: false, mergeable: bool|null}`.
     - `GET /repos/{owner}/{repo}/installation` — the App-level repo-installation lookup (pr-minder's `repoInstallationId`), registered OUTSIDE requireAuth exactly like the mint (App-JWT bearer, self-verified, app principal; `repo_installation_cache` table). Rebuild: `{id, account{login,type}, repository_selection, app_id, app_slug, target_type}`. Invalidated on `installation`/`installation_repositories` events by the stored installation id, plus a 24h TTL. 404 ("not installed") is never stored.
     Truth rows are global; the reveal layer is what keeps callers apart. Invalidation deletes are global too (a webhook is a global fact). Every response-cache table is pruned on write: expired rows plus LRU beyond `ghdata.CacheMaxRows` (10k; a var only for tests). Dispositions land in the request log: `hit` (served from state), `miss` (fetched+absorbed, with upstream status), `passthrough` (read forwarded/replayed unstored), `write` (mutation proxied), `error`.
  3. **Everything else — verbatim passthrough, uncached.** Transparently reverse-proxied to GitHub via `newGitHubProxy` (`internal/api/proxy.go`), wired as chi's `NotFound`/`MethodNotAllowed` (so the `Use` middleware — CORS, recoverer — still wraps it); the GraphQL handler forwards any non-org query the same way. The proxy forwards the caller's `Authorization` verbatim (never the server token), rejects tokenless requests with 401, targets `gh.BaseURL()` (tests hit the fake upstream), and `ModifyResponse` strips GitHub's duplicate `Access-Control-Allow-*` while keeping `Access-Control-Expose-Headers`. A 2xx passthrough under `/repos/{owner}/{repo}/...` refreshes the caller's grant (GitHub just proved access — `refreshGrantOn2xx`). Mutating methods are logged as `write`, not `passthrough` — a write is proxied because it mutates, not because caching failed. Do NOT make this path touch the freshness store.

  Adding a cached REST route means tier 2: reveal gate + global structured state + trimmed rebuild + the recursive no-URL-keys invariant test + explicit invalidation/TTL/pruning — never verbatim byte replay, and never a byte-identity requirement (that is tier 1's, only).
- **Installation tokens at rest (security note)** — the token-mint cache stores each minted installation token in SQLite (`install_token_cache.token`). This is the same trust domain as the traffic itself: the mirror already relays these exact tokens in cleartext on every request it proxies, the DB file sits on the same host with the same access, and every row is bounded by the token's own short (~1h) expiry plus the serve-until cutoff. Rows are pruned on write (expired + LRU) and flushed on installation events. Token values are never logged (the request log records method/path/disposition only) and never appear in error messages; keep it that way when touching `respcache.go`.
- **OAuth token-exchange relay** — `POST /login/oauth/access_token` (`internal/api/oauth.go`) relays a GitHub OAuth code-for-token exchange to `github.com` (NOT `api.github.com`) and returns it with the mirror's CORS headers, so a purely client-side app can complete login (GitHub's token endpoint sends no CORS headers). It is registered outside `requireAuth` (the OAuth `client_secret` in the body is the credential — no bearer token) and forwards only the body + content-negotiation headers to a fixed URL (`githubOAuthTokenURL`, a var so tests can point it at a fake). It does NOT copy GitHub's `Access-Control-*` (corsMiddleware is the single CORS authority, so no duplicate ACAO). This is why repo-nightmare's `TOKEN_PROXY` points here instead of a standalone CORS proxy.
- **Dashboard = separate authz model** — the web dashboard (`internal/api/dashboard.go`, served at `/`) authenticates a human via **GitHub OAuth** and authorizes by **login** (session cookie), which is deliberately distinct from the data API's bearer-token + reveal model. It reports the GLOBAL truth totals (one cache, one truth — same numbers for everyone) plus per-**principal** reveal-layer standing: live grant count, org-sync freshness, recent refreshes. A normal user sees their own principal(s); logins in `ADMIN_LOGINS` (default `PazerOP`) see every known principal. The `actor_identities` table maps principal→login for this labeling only. Admins additionally get: the **webhook delivery log** (`GET /api/webhooks`, "Webhooks" tab — dispositions are `applied`/`invalidated`/`ignored`/`error`; there is no `skipped`), the **request-activity log** (`GET /api/requests`, in-memory `requestLog`, "Requests" tab — `hit`/`miss`/`passthrough`/`write`/`error` with per-disposition totals), the **rate-limit view** (`GET /api/ratelimit`, "Rate limit" tab — `{live, observed, note?}`: `live` is the GitHub App's per-installation `GET /rate_limit` poll, `observed` is the `internal/ratemeter` snapshot of passively recorded `X-RateLimit-*` headers per (identity, resource); with no App configured — or a failed poll — the endpoint answers 200 with an empty `live`, a `note`, and the observed data, never a bare 503), and `GET /api/jobs` (workflow jobs; no dashboard tab yet). The request log and the rate meter are **in-memory** (live views, not audit logs) so they reset on restart.
- **Admin truth browse, grants view + consistency check (admin-only operator tooling)** — `GET /api/cache/data` dumps the GLOBAL truth rows (repos with visibility, open PRs, commit checks) as flattened JSON; `GET /api/cache/data?principal=<key>` instead dumps that principal's **access grants** (who can see what, each grant's source and expiry) — surfaced as the "Browse truth" / "Run consistency check" buttons on the admin Principals tab and a per-principal "Grants" button. `GET /api/cache/check[?org=<owner>]` runs the **consistency check** (`internal/sync/consistency.go`): it re-fetches the source of truth from GitHub via the mirror's **GitHub App** (`AppAuthenticator`, shared with background refresh) and diffs GLOBAL truth against it — one comparison for the one cache (repos/PRs only-in-cache / only-on-github, field mismatches like `last_commit_status`, `default_branch_status`, labels, draft; `mergeable` is deliberately not compared — the cache un-resolves it on pushes). A missing repo that is **private** on GitHub keeps `issue=only_on_github` but gains `visibility:"private"` + a lazy-truth note (no webhook and no principal's sync has absorbed it yet — expected, not a failure), tallied separately in `repos_only_on_github_private`; visibility comes from a checker-private GraphQL query (`ghclient.OrgRepoVisibility`), NEVER by extending the identity-locked shared org-data query. The report carries `truth_freshness`: per owner, the NEWEST org list-sync marker any principal holds (state, last fetched, error, which principal), since any principal's sync refreshes global truth. The check is **read-only**, compares **OPEN** PRs only, and only checks owners with an Organization-type App installation (others under `orgs_skipped`). Returns **503** when no App is configured.
- **TypeScript front-end** — the dashboard JS is authored in `internal/api/web/src/*.ts` and compiled by `npm run build` (tsc) to `internal/api/web/assets/*.js`, which is **committed as a generated artifact** and embedded via `//go:embed`. Edit the `.ts`, never the `.js`. CI's `web-check` job fails if the committed JS is stale (run `npm run build` and commit). `assets/demo-data.js` is preview-only (NOT embedded); the CI `preview` job injects it to deploy a backend-free styling preview to buildhost per branch. Each `src/*.ts` emits its own standalone ES module loaded by its own `<script type="module">` tag in `index.html` — `rate-meter.ts` self-registers the `<rate-meter>` web component. Adding an asset file means wiring it in **three** places: the `//go:embed` + hashed-URL machinery in `internal/api/dashboard.go`, the `index.html` script tag, and the `preview` job's copied-assets list in `ci.yml`.
- **Build** — use `go-toolchain` (not `go build`/`go test` directly); run `npm run build` after editing `web/src/*.ts`

## Environment Variables

- `GITHUB_APP_ID` (optional) — GitHub App ID. When set with a private key, the service signs in as this app for background work — periodic refreshes and the source-of-truth fetch for the admin consistency check (`/api/cache/check`); it is never used to serve normal data-API requests, which require the caller's own `Authorization` header (401 otherwise). Unset = periodic refreshes disabled and the consistency check reports "unavailable" (503). Webhooks apply to global truth either way.
- `GITHUB_APP_PRIVATE_KEY` / `GITHUB_APP_PRIVATE_KEY_PATH` — the App's PEM private key (PKCS#1 or PKCS#8), inline (optionally `\n`-escaped) or as a file path (path wins). Required when `GITHUB_APP_ID` is set.
- `WEBHOOK_SECRET` — GitHub webhook HMAC secret
- `LISTEN_ADDR` — HTTP listen address (default `:8080`)
- `DB_PATH` — SQLite database file path (default `github-mirror.db`)
- `ALLOWED_ORIGINS` (optional) — comma-separated CORS allow-list for browser clients (e.g. the repo-nightmare PR viewer). Defaults to `*` (any origin), which is safe because the cache reveals data per the caller's proven GitHub access, not origin. Preflight `OPTIONS` is answered without auth; see `internal/api/cors.go`.
- `GITHUB_OAUTH_CLIENT_ID` / `GITHUB_OAUTH_CLIENT_SECRET` (optional) — GitHub OAuth App credentials for dashboard login. If unset, the dashboard still renders but sign-in is disabled. Register the OAuth App's callback as `<BASE_URL>/auth/callback`.
- `SESSION_SECRET` (optional) — HMAC key for session cookies. If unset, a random per-process key is used (sessions reset on restart); set it in production.
- `ADMIN_LOGINS` (optional) — comma-separated logins that get the admin dashboard views (all principals, truth browse, consistency check; default `PazerOP`); case-insensitive.
- `BASE_URL` (optional) — public base URL (e.g. `https://github-state-mirror.pazer.io`) used to build the OAuth `redirect_uri`; derived from the request (honoring `X-Forwarded-Proto`) when unset.
