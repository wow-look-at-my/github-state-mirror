# GitHub State Mirror

## Project Overview

A Go service that mirrors GitHub state into SQLite, providing a fast local API surface. This is a **cache** ‚Äî the SQLite DB is disposable and gets nuked+recreated on schema version changes.

## Architecture

- `internal/actor/` ‚Äî Context-based actor (cache-partition key) propagation + display shortening (stdlib-only, safe to import from anywhere)
- `internal/freshness/` ‚Äî Generic cache freshness framework (zero GitHub knowledge)
- `internal/database/` ‚Äî SQLite schema + sqlc-generated queries (`dbgen/` is codegen, do not edit)
- `internal/ghdata/` ‚Äî Domain store wrapping sqlc with transaction logic (`dashboard.go` = cross-actor cache stats)
- `internal/ghclient/` ‚Äî GitHub REST + GraphQL API client (includes the in-memory token‚Üíidentity cache behind `ResolveTokenIdentity` and `AppAuthenticator` for GitHub App JWT / installation-token minting)
- `internal/auth/` ‚Äî GitHub OAuth login + signed-cookie sessions for the dashboard (stdlib-only, no DB knowledge)
- `internal/sync/` ‚Äî Bridge layer: concrete fetchers, periodic refresh, webhook dispatch
- `internal/webhook/` ‚Äî Webhook HTTP handler + event parsing (dispatch is **synchronous**; the handler returns a per-delivery disposition in the response so GitHub's delivery record reflects whether the cache was updated)
- `internal/api/` ‚Äî chi router, REST handlers, GraphQL cache assembler, the GitHub passthrough proxy (`proxy.go`), the in-memory request-disposition log (`requestlog.go`), and the web dashboard (`dashboard.go` + embedded `web/`)
- `internal/api/web/` ‚Äî Dashboard front-end: `src/*.ts` (TypeScript source) compiled to `assets/*.js` (committed, generated, embedded)
- `cmd/server/` ‚Äî Entry point

## Key Constraints

- **No migrations** ‚Äî bump `SchemaVersion` in `internal/database/db.go` to nuke+recreate
- **sqlc codegen** ‚Äî run `sqlc generate` after modifying `schema.sql` or `queries/*.sql`
- **Freshness/data separation** ‚Äî `internal/freshness/` must never import GitHub-specific packages
- **Lazy fetches are detached and backoff-gated** ‚Äî `freshness.Manager.doFetch` runs the fetch and its metadata writes on a `context.WithoutCancel`-derived context: a client aborting its request cannot kill the shared multi-page org-repos fetch mid-flight or prevent the result from being stored (previously an impatient browser could stop a scope from ever refreshing). The caller's explicit *deadline* (e.g. the webhook dispatch timeout) is re-applied to the fetch context, so bounded callers stay bounded. An error-state row inside its `retry_after` window is NOT re-fetched by lazy callers ‚Äî `EnsureFreshOutcome` reports the stored error instead, so a failing upstream isn't hammered on every request; deliberate refreshes (periodic/webhook triggers) bypass the backoff. When `/graphql` serves the stale cache after a refresh failure it sets `X-GSM-Stale: true` + `X-GSM-Last-Fetched` response headers and WARN-logs with the actor ‚Äî headers only, the body stays identity-test-identical.
- **Per-USER cache isolation (1 GitHub user == 1 cache scope)** ‚Äî all cached data is scoped by actor. For user tokens the actor is **`user:<numeric GitHub user id>`**: `requireAuth` (`internal/api/router.go`) resolves every bearer token via `ghclient.ResolveTokenIdentity` (`GET /user` with the caller's token, answer cached in memory per token) and all of a user's tokens ‚Äî rotating sandbox PATs, OAuth logins, narrow and broad PATs alike ‚Äî share one warm, webhook-fed bucket (the numeric id survives login renames; ids are never recycled). **Accepted trade-off (operator decision, 2026-07-03, for this instance):** any token of a user reads what any of that user's tokens cached, including private-repo data cached by a broader-scoped token; DISTINCT users remain fully isolated from each other. A token that is definitively NOT a user (GitHub answers `/user` with a non-rate-limit 403 or a 404 ‚Äî installation tokens, some CI tokens) keeps the old per-token SHA-256 **fingerprint** partition (`ghclient.Fingerprint`), with the verdict cached per token. **Fail closed:** a transient `/user` failure (network, 5xx, 429/rate-limited 403) with no cached verdict fails the request with 503 ‚Äî never a silent fingerprint fallback, since mis-partitioning a user is worse than a failed request; with a cached verdict the request is served normally. The `actor` column is part of every table's primary key. Data endpoints reject tokenless requests with 401 (and GitHub-rejected tokens ‚Äî 401 on `/user` ‚Äî with 401). There is **no static service token**: the **GitHub App** (`ghclient.AppAuthenticator`) is the service's only credential, used for background work (periodic refreshes and the webhook on-demand pull below), never to serve requests; the periodic refresher mints a short-lived access token per installation and runs in its own stable `app-installation:<id>` partition (see `sync.AppSessions`). Webhooks invalidate/apply across all actors via `MarkStaleByKindKey` / `ActorsForRepo` ‚Äî `ActorsForRepo` is `SELECT DISTINCT actor FROM repos`, so a delivery for a repo no partition has cached would otherwise be `skipped` ("no cached scope"). Instead the dispatcher **pulls on demand**: `actorsForRepo` (`internal/sync/webhook.go`), finding zero actors, mints an installation token from the delivery's `installation.id` and `EnsureFresh`es org-repos for the owner into the `app-installation:<id>` partition, then re-queries ‚Äî so the first webhook for a repo bootstraps a scope itself (one fetch seeds every repo the installation can see). This is best-effort: no app configured, no `installation.id`, or a failed fetch falls through to `skipped` (never `error`, to avoid GitHub retry storms). Do NOT replace this with proactive enumeration/seeding ‚Äî partitions populate on demand (lazy fetch + this pull); the periodic refresher only `RefreshAllOfKind`s what a partition already holds, as a backstop.
- **Webhook-fed cache (the whole point)** ‚Äî the dispatcher (`internal/sync/webhook.go`) applies payloads **directly** to the cache so high-frequency events never trigger a GitHub re-fetch: `pull_request`/`pull_request_review` upsert the PR; `status`/`check_run`/`check_suite` aggregate per-check state in the `commit_checks` table and roll it up onto each PR's `last_commit_status` (and the repo's `default_branch_status` when the check ran on the default branch, matched by `head_branch`/`branches` vs the payload's `default_branch`); `push` updates `pushed_at`; `label` recolors/removes labels. Invalidation (`MarkStaleByKindKey`) is only a **fallback** for structural events (`repository`/`organization`/`membership`) or unparseable payloads. Do NOT regress this into invalidate-and-refetch ‚Äî leveraging pushed webhook state to avoid upstream fetches is the entire purpose of the project. `UpsertPullRequest` COALESCEs `last_commit_status` AND `mergeable` against the existing row, so a PR webhook ‚Äî which carries no CI state, and carries `mergeable: null` while GitHub is still computing it (with no redelivery once it resolves) ‚Äî doesn't clobber known values; a genuinely resolved `mergeable` still overwrites. `UpsertPRForActors` additionally derives `last_commit_status` from the commit checks already recorded for the PR's head sha (per actor, from that actor's own `commit_checks` rows only), so a PR opened *after* its head commit's CI finished (pr-minder auto-opened PRs) doesn't sit at NULL forever waiting for a check event that will never re-fire. The handler dispatches **synchronously** (the cache writes are fast, idempotent upserts) and returns a `DispatchResult` whose disposition ‚Äî `applied` (200) / `skipped`¬∑`ignored`¬∑`invalidated` (202) / `error` (500) ‚Äî is the response GitHub records; every delivery is also written to the global `webhook_deliveries` log. Don't restore the old fire-and-forget 200: it made every delivery look successful regardless of whether data was preserved. The dispatcher also maintains the trimmed response caches (disposition-neutral, best-effort): `push`/`repository` flush the repo's `contents_cache` rows, `installation`/`installation_repositories` flush the installation's `install_token_cache` rows (`invalidateResponseCaches`), and `onPush` absorbs the payload's commits into `git_commits_cache` (`absorbPushCommits`) ‚Äî see the cache-contract bullet below. `workflow_job` deliveries (`in_progress` and `completed` only; `queued`/`waiting` churn is dropped as `ignored`) record GitHub Actions job state in the **global** `workflow_jobs` table ‚Äî global like `webhook_deliveries`, NOT actor-scoped, because it is webhook-fed operational telemetry with no per-credential fetch path (a job's state only ever arrives via the HMAC-verified delivery, never through a caller's token; read via the admin-only `GET /api/jobs`, running jobs first then completed, `limit` default 100 / cap 500). The upsert (`UpsertWorkflowJob`) is out-of-order tolerant: a `completed` arriving before its `in_progress` still records the full row, and a late `in_progress` never regresses an already-completed row (CASE guards on the existing status). Growth is bounded by pruning on write ‚Äî after each upsert, completed jobs older than 14 days (`ghdata.workflowJobRetention`, a code constant on purpose) are removed in one indexed DELETE (partial index `idx_workflow_jobs_completed_at`); in-progress rows are never pruned. **Operator action required**: the GitHub App must be **subscribed to the `workflow_job` event** in its settings or the mirror receives nothing ‚Äî expected volume is high in CI-heavy orgs, and the per-delivery handler cost is one cheap synchronous SQLite upsert.
- **App-identity partition (opt-in, for trusted first-party app callers)** ‚Äî a *request*-time caller may send a **GitHub App JWT** in the `X-Mirror-Identity` header. `requireAuth` verifies it via `ghclient.VerifyAppIdentity` (`GET /app` ‚Äî GitHub only 200s if the RS256 signature checks out against the app's public key, so it's unforgeable; cached per-JWT) and partitions that caller as `app:<id>`, skipping the per-user/fingerprint resolution entirely (installation tokens cannot call `/user`). This is distinct from the background refresher's `app-installation:<id>` partitions above: that's the mirror signing in as *its own* app for periodic refreshes; this is an *external* app caller (e.g. pr-minder) tagging its data-API requests. It exists because such a caller's installation tokens rotate hourly ‚Äî fingerprinting them would give a fresh empty bucket every hour; an app identity gives **one stable, webhook-fed bucket** for all the app's tokens. The `Authorization` token is still injected into the context for upstream fetches/passthrough, so per-repo authorization is unchanged. Security caveat: within an app's bucket, any of that app's tokens can read what the bucket cached (the reader is a single trusted codebase), so this is appropriate **only** for a first-party app, never as a way to relax the default per-user/fingerprint isolation. No identity header ‚Üí the default per-user/fingerprint behavior above. (Without the header an installation token now lands on its fingerprint partition ‚Äî 403 on `/user` is the cached not-a-user verdict ‚Äî instead of being 401-rejected as it was before per-user partitioning.)
- **The cache contract (three tiers)** ‚Äî every data route falls into exactly one of these; know which tier a route is in before touching it:
  1. **GraphQL org-repos ‚Äî byte-identical, identity-test-locked (unchanged).** `POST /graphql`'s org-repos query is assembled in `graphql.go` to match exactly GitHub's selection set, locked by an identity test (`identity_shape_test.go`). This is the ONLY route where byte-identity to GitHub applies. Any other GraphQL query forwards to the passthrough.
  2. **Cached REST routes ‚Äî state absorbed, rebuilt TRIMMED (operator decision 2026-07-03).** `internal/api/respcache.go` + `internal/ghdata/respcache.go` + the `contents_cache`/`git_commits_cache`/`install_token_cache` tables. The route ABSORBS the state a GitHub response contains into structured columns and REBUILDS the response from that state, **dropping every URL field**: `url`, anything matching `*_url` (`html_url`, `git_url`, `download_url`, `documentation_url`, ...), and `_links`. GitHub REST responses are full of URL clutter; the mirror's consumers are first-party (pr-minder etc.) and read state fields only, so cached routes must NOT emit them ‚Äî verbatim byte caching was rejected outright. (Yes, this deliberately brings the old "trimmed shapes" model back for these routes ‚Äî but now as an explicit, documented contract with a locked no-URL-keys invariant test, `TestRespCache`/`assertNoURLKeys` in `respcache_test.go`, not an accident.) Hits AND misses both serve the rebuilt shape (a route's shape never flip-flops with cache state), marked `X-GSM-Cache: hit|miss`; rebuilt responses carry only `Content-Type: application/json; charset=utf-8` (no replayed ETag/rate-limit/request-id headers ‚Äî those describe GitHub's handling of one request, which a rebuild is not). Anything a route cannot model passes through verbatim, unstored, recorded as a `passthrough`: non-default `Accept` media types (raw/html/object change the shape), unknown query params, symlink/submodule objects, >1 MiB `"encoding":"none"` files, oversized bodies, unexpected statuses. The v1 routes:
     - `GET /repos/{owner}/{repo}/contents/{path...}` (incl. `?ref=`, part of the key) ‚Äî absorbs 200 file (base64), 200 dir (trimmed entries), **and 404** (the "config file absent" answer is half the win; rebuilt `{"message": ..., "status": "404"}`). Invalidated on `push`/`repository` webhooks ‚Äî conservative whole-repo flush (`InvalidateContentsCache`; the push payload's modified-paths refinement can come later) ‚Äî plus a **24h TTL backstop** (`contentsCacheTTL`) so a missed webhook can't serve stale forever.
     - `GET /repos/{owner}/{repo}/git/commits/{sha}` ‚Äî immutable: no invalidation, no TTL, LRU-bounded only. Rebuild: `{sha, author{name,email,date}, committer{...}, message, tree:{sha}, parents:[{sha},...]}`. **Push payloads feed this cache too** (`absorbPushCommits` in `internal/sync/webhook.go`): each pushed commit's id/tree_id/message/timestamp/author/committer is upserted for every actor with the repo cached, so pr-minder's post-push read hits with zero GitHub fetches. Parents come from the payload's linear chain (`webhook.PushPayload.ChainedCommits`: before ‚Üí commits[0] ‚Üí ...), which DECLINES forced pushes, new refs (all-zeros before), possibly-truncated arrays (‚â•20), and non-linear orderings rather than derive wrong parents; a pushed merge commit's extra parents are the one undetectable case, acceptable because the only parents-reading consumer (pr-minder's test-merge check) reads them off refs/pull/N/merge commits, which never arrive via push payloads. Webhook-sourced and fetch-sourced rows rebuild to the same shape. 404s are never cached here (a missing sha can be pushed later).
     - `POST /app/installations/{id}/access_tokens` ‚Äî the token-mint cache, registered OUTSIDE requireAuth (an App JWT cannot resolve `GET /user`): the handler verifies the bearer itself via `ghclient.VerifyAppIdentity` (the same unforgeable `GET /app` check the `X-Mirror-Identity` header uses) and partitions as `app:<id>`. Key: (app id, installation id, SHA-256 of the **canonicalized** request body ‚Äî whitespace-insensitive, JSON keys sorted; an empty body vs a permissions/repositories subset are DIFFERENT tokens). Only a 201 whose `expires_at` parses is stored, served until `expires_at - 10min` (`mintExpiryBuffer`); rebuild is exactly `{token, expires_at, permissions?, repository_selection?}` (GitHub's `repositories` array ‚Äî full repo objects, URLs and all ‚Äî is dropped). Invalidated on `installation`/`installation_repositories` events for that installation (suspend/delete must not keep serving). An unverifiable bearer forwards unchanged, uncached.
     `GET /repos/{o}/{r}/pulls/{n}` is deliberately NOT cached in v1: its body carries the lazily-computed `mergeable` field and pr-minder polls that endpoint waiting for `mergeable` to resolve ‚Äî a cached body would wedge the poll. Leave it passthrough. Cached-route reads are partitioned by the SAME actor as everything else (requireAuth's actor; the contents/git-commits routes sit inside the requireAuth group, so pr-minder's `X-Mirror-Identity` calls share the `app:<id>` bucket across its rotating tokens). Invalidation deletes span ALL actors (a webhook is a global fact). Every table is pruned on write: expired rows plus LRU beyond `ghdata.CacheMaxRows` (10k; a var only for tests). Dispositions land in the request log: `hit` (served from state), `miss` (fetched+absorbed, with upstream status), `passthrough` (forwarded/replayed unstored) ‚Äî so the dashboard's counters reflect REST traffic for real. Webhook wiring is `invalidateResponseCaches` at the top of the dispatcher's `handle` ‚Äî deliberately **disposition-neutral** (best-effort, alongside the existing apply logic; it never changes what a delivery reports).
  3. **Everything else ‚Äî verbatim passthrough, uncached.** Transparently reverse-proxied to GitHub via `newGitHubProxy` (`internal/api/proxy.go`), wired as chi's `NotFound`/`MethodNotAllowed` (so the `Use` middleware ‚Äî CORS, recoverer ‚Äî still wraps it); the GraphQL handler forwards any non-org query the same way. The proxy forwards the caller's `Authorization` verbatim (never the server token), rejects tokenless requests with 401, targets `gh.BaseURL()` (tests hit the fake upstream), and `ModifyResponse` strips GitHub's duplicate `Access-Control-Allow-*` while keeping `Access-Control-Expose-Headers`. Do NOT make this path touch the freshness store.

  Adding a cached REST route means tier 2: structured state + trimmed rebuild + the recursive no-URL-keys invariant test + explicit invalidation/TTL/pruning ‚Äî never verbatim byte replay, and never a byte-identity requirement (that is tier 1's, only).
- **Installation tokens at rest (security note)** ‚Äî the token-mint cache stores each minted installation token in SQLite (`install_token_cache.token`). This is the same trust domain as the traffic itself: the mirror already relays these exact tokens in cleartext on every request it proxies, the DB file sits on the same host with the same access, and every row is bounded by the token's own short (~1h) expiry plus the serve-until cutoff. Rows are pruned on write (expired + LRU) and flushed on installation events. Token values are never logged (the request log records method/path/disposition only) and never appear in error messages; keep it that way when touching `respcache.go`.
- **OAuth token-exchange relay** ‚Äî `POST /login/oauth/access_token` (`internal/api/oauth.go`) relays a GitHub OAuth code-for-token exchange to `github.com` (NOT `api.github.com`) and returns it with the mirror's CORS headers, so a purely client-side app can complete login (GitHub's token endpoint sends no CORS headers). It is registered outside `requireAuth` (the OAuth `client_secret` in the body is the credential ‚Äî no bearer token) and forwards only the body + content-negotiation headers to a fixed URL (`githubOAuthTokenURL`, a var so tests can point it at a fake). It does NOT copy GitHub's `Access-Control-*` (corsMiddleware is the single CORS authority, so no duplicate ACAO). This is why repo-nightmare's `TOKEN_PROXY` points here instead of a standalone CORS proxy.
- **Dashboard = separate authz model** ‚Äî the web dashboard (`internal/api/dashboard.go`, served at `/`) authenticates a human via **GitHub OAuth** and authorizes by **login** (session cookie), which is deliberately distinct from the data API's bearer-token + per-user partition model. For a normal user it never serves another user's cached rows ‚Äî it only reports per-scope **counts + freshness metadata**. A user sees the scopes their own tokens populated (post-per-user-partitioning that is normally the single `user:<id>` scope); logins in `ADMIN_LOGINS` (default `PazerOP`) see all scopes. The `actor_identities` table maps actor‚Üílogin for this grouping/`is_self` labeling only; data tables stay keyed by the partition key `requireAuth` resolved. (The data partition IS now keyed by the user ‚Äî deliberately: the operator decided 2026-07-03 that 1 user == 1 scope, accepting that a narrow token of a user reads what that user's broader tokens cached. It is keyed by the immutable numeric id, not the rename-able login, and only for tokens GitHub confirms belong to that user ‚Äî machine tokens stay fingerprint-isolated, app paths unchanged.) Identity rows are written (debounced) in `requireAuth` for user tokens only (a non-user token has no login). Admins additionally get two global activity logs: a **webhook delivery log** (`GET /api/webhooks`, the `webhook_deliveries` table, "Webhooks" tab) showing each delivery's disposition + detail, and a **request-activity log** (`GET /api/requests`, the in-memory `requestLog` in `requestlog.go`, "Requests" tab) showing each data-API request's cache disposition ‚Äî `hit` (served fresh) / `miss` (fetched then cached) / `passthrough` (forwarded uncached) ‚Äî plus per-disposition totals, so you can see how much the cache is actually used vs. proxied through. Disposition comes from `freshness.Manager.EnsureFreshOutcome` for the cached `/graphql` route and from the `recordPassthrough` wrapper around `newGitHubProxy` for everything forwarded; the actor label is best-effort (the resolved actor, else the `X-Mirror-Identity` app id, else a token fingerprint). Both logs span every repo and are **admin-only**; the request log is **in-memory** (request traffic is high-volume and this is a live view, not an audit log) so it resets on restart. A third admin-only global endpoint, `GET /api/jobs`, returns the recorded GitHub Actions jobs (see the `workflow_jobs` notes in the webhook-fed bullet); it has **no dashboard tab yet** ‚Äî the JSON shape is kept tab-friendly for a later front-end change.
- **Admin cache browse + consistency check (admin-only operator tooling)** ‚Äî two admin-gated endpoints (`requireAdmin` in `internal/api/browse.go`, surfaced as per-scope **Browse** / **Check** buttons in the dashboard) let the operator inspect a scope beyond counts. `GET /api/cache/data?actor=<fp>` dumps the actual cached rows for one scope as flattened JSON (`collectBrowse` + the `*ByActor` reads in `internal/ghdata/browse.go` / new `dashboard.sql` queries). `GET /api/cache/check?actor=<fp>[&org=]` runs the **consistency check** (`internal/sync/consistency.go`): it re-fetches the source of truth from GitHub via the mirror's **GitHub App** (`AppAuthenticator`, shared with background refresh ‚Äî `buildAppAuthenticator` in `main.go` builds it once and feeds both the periodic `SessionFunc` and the `ConsistencyChecker`) and emits a JSON diff (repos/PRs only-in-cache / only-on-github, plus field mismatches like `last_commit_status`, `default_branch_status`, labels, draft). A missing repo that is **private** on GitHub keeps `issue=only_on_github` (stable for tooling) but gains `visibility:"private"` + a note that the scope's token may simply not see it, tallied separately in `repos_only_on_github_private` ‚Äî visibility comes from a checker-private GraphQL query (`ghclient.OrgRepoVisibility`, best-effort), NEVER by extending the identity-locked shared org-data query. The report header also carries `scope_freshness`: per owner, the scope's own org_repos cache metadata (fetch state, last_fetched_at, last error) read from the freshness store (wired into `NewConsistencyChecker`). The check is **read-only** (never writes the cache), only compares **OPEN** PRs (the cache only retains open PRs), and only checks owners with an Organization-type App installation (others go under `orgs_skipped`, not reported as missing). Returns **503** when no App is configured. This does NOT relax partition isolation ‚Äî it just lets the operator read what is already cached and diff it; it is gated to `ADMIN_LOGINS`. `scopeStats.actor_id` carries the full partition key (`user:<id>`, a hex fingerprint, or `app-installation:<id>`) so the admin UI can target a scope; display uses `actor.Short`, which abbreviates only opaque hex fingerprints and shows structured actors whole.
- **TypeScript front-end** ‚Äî the dashboard JS is authored in `internal/api/web/src/*.ts` and compiled by `npm run build` (tsc) to `internal/api/web/assets/*.js`, which is **committed as a generated artifact** and embedded via `//go:embed`. Edit the `.ts`, never the `.js`. CI's `web-check` job fails if the committed JS is stale (run `npm run build` and commit). `assets/demo-data.js` is preview-only (NOT embedded); the CI `preview` job injects it to deploy a backend-free styling preview to buildhost per branch. Each `src/*.ts` emits its own standalone ES module loaded by its own `<script type="module">` tag in `index.html` ‚Äî `rate-meter.ts` self-registers the `<rate-meter>` web component (shadow DOM owns the tile's markup/styles; themed via the inherited `:root` custom properties) that the rate-limit tab's tiles are built from. Adding an asset file means wiring it in **three** places: the `//go:embed` + hashed-URL machinery in `internal/api/dashboard.go`, the `index.html` script tag, and the `preview` job's copied-assets list in `ci.yml`.
- **Build** ‚Äî use `go-toolchain` (not `go build`/`go test` directly); run `npm run build` after editing `web/src/*.ts`

## Environment Variables

- `GITHUB_APP_ID` (optional) ‚Äî GitHub App ID. When set with a private key, the service signs in as this app for background work ‚Äî periodic refreshes, the webhook dispatcher's on-demand repo pulls, and the source-of-truth fetch for the admin consistency check (`/api/cache/check`); it is never used to serve normal data-API requests, which require the caller's own `Authorization` header (401 otherwise). Unset = periodic refreshes and on-demand pulls disabled (webhooks for uncached repos skip) and the consistency check reports "unavailable" (503).
- `GITHUB_APP_PRIVATE_KEY` / `GITHUB_APP_PRIVATE_KEY_PATH` ‚Äî the App's PEM private key (PKCS#1 or PKCS#8), inline (optionally `\n`-escaped) or as a file path (path wins). Required when `GITHUB_APP_ID` is set.
- `WEBHOOK_SECRET` ‚Äî GitHub webhook HMAC secret
- `LISTEN_ADDR` ‚Äî HTTP listen address (default `:8080`)
- `DB_PATH` ‚Äî SQLite database file path (default `github-mirror.db`)
- `ALLOWED_ORIGINS` (optional) ‚Äî comma-separated CORS allow-list for browser clients (e.g. the repo-nightmare PR viewer). Defaults to `*` (any origin), which is safe because data is partitioned by the caller's resolved GitHub user (or token), not origin. Preflight `OPTIONS` is answered without auth; see `internal/api/cors.go`.
- `GITHUB_OAUTH_CLIENT_ID` / `GITHUB_OAUTH_CLIENT_SECRET` (optional) ‚Äî GitHub OAuth App credentials for dashboard login. If unset, the dashboard still renders but sign-in is disabled. Register the OAuth App's callback as `<BASE_URL>/auth/callback`.
- `SESSION_SECRET` (optional) ‚Äî HMAC key for session cookies. If unset, a random per-process key is used (sessions reset on restart); set it in production.
- `ADMIN_LOGINS` (optional) ‚Äî comma-separated logins that may view **all** cache scopes (default `PazerOP`); case-insensitive.
- `BASE_URL` (optional) ‚Äî public base URL (e.g. `https://github-state-mirror.pazer.io`) used to build the OAuth `redirect_uri`; derived from the request (honoring `X-Forwarded-Proto`) when unset.
